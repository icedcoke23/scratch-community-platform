# 🏗️ 系统设计草案 v0.2（深度优化版）

> 日期：2026-04-23 | 基于 v0.1 架构审计修正
> 聚焦 Scratch，课后教学场景
> 架构：模块化单体 + 2 独立服务
> ⚠️ 本文档核心架构设计仍然有效，但数据库/搜索/部署细节已对齐 v0.4（14模块、MySQL+Redis+MinIO MVP、三级判题降级）

## 一、产品定位

**一句话**：面向 K12 课后教学的 Scratch 编程社区，融合「创作 + 社区 + 刷题 + 作业 + 积分 + AI」。

### 核心差异化

1. **sb3 深度解析** — 自动解压 → 解析完整 JSON → 提取程序链/角色/素材/变量/广播 → AI/教师分析
2. **AI 原生** — AI Gateway 统一调度：点评/判题/出题/学情分析
3. **课后教学闭环** — 作业布置 → 提交 → 自动+手动批改 → 数据统计 → 学情报告
4. **班级宠物积分** — 趣味化激励体系

## 四、sb3 解析库 — 详细设计

### 4.1 定位：共享库，不是服务

```java
// 使用方式 — 一行代码调用
SB3ParseResult result = SB3Parser.parse(sb3FileInputStream);

// 或从已存储的解析结果读取
SB3ParseResult result = SB3Parser.fromMongo(parseResultId);
```

### 4.2 解析流程

```
输入: .sb3 文件 (InputStream)

Step 1: SB3Unzipper.unzip(inputStream)
  → project.json (String)
  → assets/ (Map<String, byte[]> 素材文件)

Step 2: ProjectJsonParser.parse(projectJson)
  → targets: List<Target> (舞台 + 角色)
  → monitors: List<Monitor>
  → extensions: List<String>

Step 3: 并行提取
  ├── SpriteExtractor.extract(targets)
  │   → List<SpriteInfo> (角色信息)
  ├── VariableExtractor.extract(targets)
  │   → List<VariableInfo> (变量/列表)
  ├── BroadcastAnalyzer.analyze(targets)
  │   → List<BroadcastRelation> (广播关系)
  └── BlockChainBuilder.build(targets)
      → List<ScriptChain> (程序链)

Step 4: ComplexityCalculator.calculate(allData)
  → ComplexityScore (复杂度评分)

Step 5: 输出
  → SB3ParseResult (完整结果，存 MySQL JSON 字段，Phase 3+ 可迁移至 MongoDB)
  → SB3ParseSummary (摘要，存 MySQL)
```

### 4.3 核心数据结构

```java
@Data
public class SB3ParseResult {
    private String id;
    private ProjectMeta meta;           // 项目元信息
    private StageInfo stage;            // 舞台信息
    private List<SpriteInfo> sprites;   // 角色列表
    private List<ScriptChain> scripts;  // 程序链列表
    private List<VariableInfo> variables; // 全局变量
    private List<ListInfo> lists;       // 全局列表
    private List<BroadcastRelation> broadcasts; // 广播关系
    private List<AssetInfo> assets;     // 素材清单
    private BlockTypeStats blockStats;  // 积木类型统计
    private ComplexityScore complexity; // 复杂度评分
}

@Data
public class ScriptChain {
    private String triggerType;    // "greenflag" / "keypress" / "clicked" ...
    private String spriteName;     // 所属角色 ("_stage_" 表示舞台)
    private BlockNode root;        // 积木树根节点
    private int blockCount;        // 积木数量
    private int maxDepth;          // 最大嵌套深度
    private List<String> blockTypes; // 使用的积木类型
}

@Data
public class BlockNode {
    private String opcode;         // 积木类型 (e.g. "motion_movesteps")
    private Map<String, Object> inputs;   // 输入参数
    private Map<String, Object> fields;   // 字段值
    private BlockNode next;        // 下一个积木
    private List<BlockNode> children; // 子积木 (循环体/条件分支)
}

@Data
public class ComplexityScore {
    private int totalBlocks;       // 总积木数
    private int scriptCount;       // 程序链数
    private int maxNestingDepth;   // 最大嵌套深度
    private int uniqueBlockTypes;  // 使用的不同积木类型数
    private int variableCount;     // 变量数
    private int listCount;         // 列表数
    private int broadcastCount;    // 广播数
    private double score;          // 综合评分 (0-100)
    private Map<String, Integer> blockTypeDistribution; // 各类型积木数
}
```

| 调用方 | 调用时机 | 用途 |

## 五、判题沙箱服务 — 详细设计

### 5.1 定位：独立服务，Docker 隔离

```
判题沙箱服务 (Node.js + Express)
  ├── 接收判题请求 (HTTP)
  ├── 从队列取任务 (MQ)
  ├── 创建 Docker 容器
  │   ├── 镜像: scratch-vm-runner (预装 scratch-vm)
  │   ├── 资源限制: CPU 1核 / 内存 512MB / 超时 30s
  │   ├── 网络隔离: 无网络访问
  │   └── 文件系统: 只读 + 临时写入区
  ├── 注入测试用例 → 运行 scratch-vm → 捕获输出
  ├── 比对结果 → 返回判题结果
  └── 销毁容器
```

### 5.2 API 接口

```yaml
POST /judge/submit
  Request:
    projectUrl: string      # sb3 文件 URL
    problemId: number       # 题目 ID
    testCases: array        # 测试用例 [{input, expectedOutput}]
    timeLimit: number       # ms
    memoryLimit: number     # MB
  Response:
    result: "AC" | "WA" | "TLE" | "MLE" | "RE"
    score: number
    details: [{testCaseId, input, expected, actual, passed, timeUsed}]
    executionLog: string

POST /judge/creative
  Request:
    projectUrl: string      # sb3 文件 URL
    parseResultId: string   # sb3 解析结果 ID
    rubric: object          # 评分标准
  Response:
    score: number
    dimensions: [{name, score, comment}]
    aiComment: string
```

### 5.3 scratch-vm Headless 运行

```javascript
// scratch-vm-runner 核心逻辑
const vm = new VirtualMachine();
const projectData = fs.readFileSync(projectPath);

// 加载项目
await vm.loadProject(projectData);

// 注入测试用例 (替换"询问并等待"的回答)
vm.runtime.on('QUESTION', (question) => {
  const answer = testCases[currentCase].input;
  vm.runtime.emit('ANSWER', answer);
});

// 运行
vm.greenFlag();

// 等待执行完成或超时
await waitForCompletion(vm, timeLimit);

// 捕获输出
const output = captureOutput(vm);
// output: { variables: {...}, sprites: [...], messages: [...] }
```

## 六、AI Gateway 服务 — 详细设计

### 6.1 定位：独立服务，统一 AI 调度

```
AI Gateway (Python FastAPI)
  ├── Prompt 模板管理 (数据库存储)
  ├── 请求队列 (异步处理)
  ├── 响应缓存 (相同输入不重复调用)
  ├── 成本控制 (每用户/每日限额)
  ├── 限流 (QPS 控制)
  ├── 降级策略 (AI 不可用时返回模板)
  ├── 质量评估 (置信度过滤)
  └── 多模型路由 (OpenAI / 通义千问 / 本地模型)
```

## 七、统一判题核心 — 详细设计

### 7.1 判题流程

```
输入: submission (sb3_url + problem_id)

Step 1: 获取题目信息
  → problem = problemMapper.getById(problemId)

Step 2: sb3 解析
  → parseResult = SB3Parser.parse(sb3InputStream)

Step 3: 根据题目类型选择判题策略
  ├── 选择题: ChoiceJudge.judge(answer, problem.answer)
  ├── 填空题: FillJudge.judge(answer, problem.answer)
  ├── Scratch 算法题:
  │   → 调用判题沙箱
  │   → SandboxClient.submit(sb3Url, testCases)
  │   → 等待结果
  │   → ResultParser.parse(response)
  └── Scratch 创意题:
      → 调用 AI Gateway
      → AiGateway.creativeJudge(parseResultId, rubric)
      → 返回评分

Step 4: 分数计算
  → ScoreCalculator.calculate(result, problem.scoreRule)

Step 5: 更新状态
  → submission.result = result
  → submission.score = score
  → 更新排行榜/积分/错题本
```

## 十、搜索架构（v0.4 更新 — 分阶段策略）

```
Phase 1-2（MVP）: MySQL FULLTEXT + ngram

works 搜索:
  ALTER TABLE project ADD FULLTEXT INDEX idx_search (title, description) WITH PARSER ngram;
  SELECT * FROM project WHERE MATCH(title, description) AGAINST('关键词' IN BOOLEAN MODE);

problems 搜索:
  ALTER TABLE problem ADD FULLTEXT INDEX idx_search (title, description) WITH PARSER ngram;

users 搜索:
  -- 简单 LIKE 搜索，数据量小够用
  SELECT * FROM user WHERE nickname LIKE '%关键词%';

优势: 零额外组件，ngram 支持中文分词
限制: 不支持聚合分析、同义词、向量搜索

---

Phase 3+（按需引入 Elasticsearch 8）:

ES 索引设计:
  works_index (作品搜索):
    mappings:
      title: text (ik_max_word)
      description: text (ik_max_word)
      tags: keyword[]
      user_id: keyword
      project_type: keyword
      like_count: integer
      created_at: date
      complexity_score: float

  problems_index (题目搜索):
    mappings:
      title: text (ik_max_word)
      description: text (ik_max_word)
      tags: keyword[]
      difficulty: keyword
      problem_type: keyword
      ac_rate: float

  users_index (用户搜索):
    mappings:
      nickname: text (ik_max_word)
      bio: text
      level: integer

  同步策略:
    MySQL 写入 → Binlog → Canal → ES (近实时同步)
```

## 十一、积分防刷引擎

```
防刷规则引擎:

rule 1: 每日积分上限
  └── 每用户每天获取积分 ≤ 200

rule 2: 重复提交检测
  └── 同一作品 24h 内重复提交不计分

rule 3: 点赞唯一性
  └── 同一用户对同一作品只能点赞 1 次

rule 4: 新用户限制
  └── 注册 < 24h 的用户，积分获取减半

rule 5: 操作频率限制
  └── 同一操作 10s 内不重复计分

rule 6: 异常检测
  └── 5 分钟内积分变动 > 100 → 触发验证码

rule 7: 班级宠物防刷
  └── 积分来源可追溯，异常积分可回滚

实现:
  Redis Lua 脚本保证原子性
  异步日志记录所有积分变动
  定时任务扫描异常模式
```
