# 💡 2026-04-23 架构审计讨论

> 参与者：用户 + AI 助手
> 主题：对系统设计草案 v0.1 的深度审计

---

## 审计方法

结合市面成熟系统（Teaching-Open、编程猫、LeetCode、好学好教 OJ）的架构实践，对照我们的设计草案，从架构原则、开发效率、运维复杂度三个维度审查。

## 核心发现

### 🔴 严重问题（必须修正）

1. **微服务过度拆分**：13 个服务对 MVP 来说太多。市面成功案例（Teaching-Open、好学好教）都是单体起步。改为 1 个模块化单体 + 2 个独立服务（判题沙箱、AI Gateway）。

2. **sb3 解析不是服务**：它是纯计算逻辑，应该是共享库（SDK），不是独立微服务。被 editor/social/judge/homework/analytics 各模块直接调用。

3. **OJ 与作业的判题核心重复**：两者的判题逻辑完全相同，只是业务场景不同。统一为 judge-core 库，被不同业务模块包装。

### 🟡 中等问题

4. **AI 服务设计太粗略**：缺少 Prompt 管理、成本控制、缓存、降级、质量评估等关键设计。需要 AI Gateway 层。

5. **缺少实时通知**：作业批改完成/被点赞/宠物升级等场景需要 WebSocket 推送。

6. **内容审核流程缺失**：需要三级审核流水线（规则 → AI → 人工）。

7. **搜索无具体设计**：ES 索引需要明确 schema。

8. **班级宠物防刷缺失**：需要多层防刷机制。

### 🟢 优化建议

9. 增加 CDN 和静态资源架构
10. 增加数据埋点和分析流水线
11. Scratch 编辑器按需加载策略
12. 预埋多租户字段（tenant_id）

## 修正后的架构

```
部署单元 = 3 个:
  1. 模块化单体 (Spring Boot 3) — 包含所有业务模块
  2. 判题沙箱 (Docker 容器池) — 安全隔离执行用户代码
  3. AI Gateway — LLM 调用/缓存/降级/成本控制
```
