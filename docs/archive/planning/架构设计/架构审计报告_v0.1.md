# 🔬 架构深度审计报告

> 日期：2026-04-23
> 审计对象：系统设计草案 v0.1
> 方法：对比市面成熟系统 + 架构原则审查

---

## 审计结论（先说结论）

当前设计有 **3 个严重问题**、**5 个中等问题**、**4 个优化建议**。核心问题是：**微服务过度拆分**、**sb3 解析定位模糊**、**AI 能力设计过于粗略**。

---

## 🔴 严重问题

### 问题 1：微服务过度拆分 — 13 个服务太多了

**现状**：设计了 13 个独立微服务（user/editor/social/sb3-parser/problem/judge/homework/points/ai/analytics/file/msg/gateway）。

**为什么是问题**：

| 维度 | 影响 |
|---|---|
| 开发成本 | 13 个服务 = 13 套配置、13 个 Dockerfile、13 套 CI/CD、13 个数据库 schema |
| 运维复杂度 | 服务间调用链长，排错困难，需要分布式追踪 |
| 团队规模 | 如果团队 < 10 人，13 个服务根本维护不过来 |
| 数据一致性 | 跨服务事务极其复杂（如：提交作业同时要更新积分、更新班级宠物、更新排行榜） |

**市面参照**：
- **Teaching-Open**：单体 Spring Boot，所有功能在一个应用里，开发部署极其简单
- **编程猫**：虽然是微服务，但初期也是单体起步，逐步拆分
- **LeetCode**：至今核心判题+社区仍然在同一个应用内
- **好学好教 OJ**：单体架构，够用就行

**正确做法**：**单体优先 (Monolith First)**

```
阶段 1 (MVP): 1 个后端应用 + 1 个前端
  └── 模块化单体:
       ├── user 模块
       ├── editor 模块
       ├── social 模块
       ├── sb3-parser 模块 (作为库)
       ├── problem + judge 模块
       ├── homework 模块
       ├── points 模块
       └── analytics 模块

阶段 2 (规模化): 按性能瓶颈拆分
  └── 只把真正需要独立扩展的拆出来:
       ├── 判题引擎 (需要独立沙箱，CPU 密集)
       ├── AI 服务 (外部 API 调用，I/O 密集)
       └── 其余保留在单体中

阶段 3 (大规模): 按团队边界拆分
  └── 团队 > 10 人时再考虑全面微服务
```

**建议**：将 13 个服务合并为 **1 个模块化单体 + 2 个独立服务**（判题引擎、AI 服务）。

### 问题 2：sb3 解析不应是独立微服务

**现状**：`sb3-parser-svc` 被设计为一个独立微服务，有自己的数据库。

**为什么是问题**：

1. **它不是服务，它是库**：sb3 解析是纯计算逻辑（解压 ZIP → 解析 JSON → 输出结构），不需要独立进程、独立端口、独立数据库
2. **性能浪费**：每次解析要走一次网络调用，比本地函数调用慢 10-100 倍
3. **部署复杂**：多一个服务 = 多一个需要监控/重启/扩缩的单元

**正确做法**：

```
sb3-parser 应该是一个共享库 (SDK/Library):
  ├── npm 包 (Node.js 版，用于前端预览解析)
  ├── Java 库 (后端版，用于判题/批改)
  └── 被以下模块直接调用:
       ├── judge 模块: 判题时调用 parser 解析 sb3
       ├── homework 模块: 提交作业时调用 parser 解析
       ├── social 模块: 发布作品时调用 parser 提取封面/信息
       └── analytics 模块: 统计时调用 parser 分析编程习惯
```

**例外**：如果解析量极大（每天百万级），可以考虑将解析任务放到消息队列异步处理，但仍然不需要独立服务——放在 worker 进程里即可。

### 问题 3：OJ 判题与作业批改的边界不清

**现状**：OJ（`judge-svc`）和作业（`homework-svc`）是两个完全独立的服务。

**为什么是问题**：

| 场景 | 实际行为 |
|---|---|
| 作业里的编程题 | 提交 sb3 → 需要判题 → 调用 judge-svc？ |
| OJ 里的题目 | 提交 sb3 → 判题 → 和作业有什么区别？ |
| 测评里的题目 | 提交 sb3 → 判题 → 又是另一套？ |

本质上，**OJ 判题、作业批改、测评评分都用同一套判题引擎**，区别只在于：
- 谁发起的（学生自主 vs 教师布置）
- 分数怎么用（积分 vs 成绩）
- 截止时间（无 vs 有）

**正确做法**：

```
统一判题核心 (judge-core):
  ├── 标准判题 (输出比对)
  ├── 创意题判题 (AI 评分)
  └── 被以下场景复用:
       ├── OJ 刷题: 学生自主 → judge-core → 积分
       ├── 作业批改: 教师布置 → judge-core → 成绩
       └── 测评定时: 定时触发 → judge-core → 测评分
```

## 🟡 中等问题

### 问题 4：AI 服务设计过于粗略

**现状**：AI 服务的描述是 "调用外部 LLM"，没有具体架构。

**缺失的关键设计**：

| 缺失 | 为什么重要 |
|---|---|
| Prompt 管理 | 不同场景（点评/判题/出题）需要不同的 Prompt，如何管理？ |
| 成本控制 | LLM API 调用很贵，如何限制频率/缓存结果？ |
| 响应时间 | LLM 调用需要 3-30 秒，用户体验如何保证？ |
| 降级策略 | AI 挂了怎么办？如何 fallback？ |
| 质量评估 | AI 点评的质量如何保证？需要人工审核吗？ |
| 数据隐私 | 学生作品数据发给第三方 LLM 是否合规？ |

**建议的 AI 服务架构**：

```
AI Gateway (统一入口)
  ├── Prompt 模板管理 (数据库存储各场景 Prompt)
  ├── 请求队列 (异步处理，避免阻塞)
  ├── 缓存层 (相同作品不重复调用)
  ├── 成本控制 (每用户/每日 API 调用上限)
  ├── 降级策略 (AI 不可用时返回模板点评)
  ├── 质量评估 (AI 输出置信度，低置信度转人工)
  └── 多模型支持 (OpenAI / 国产大模型 / 本地模型切换)
```

### 问题 5：缺少实时通知架构

**现状**：`msg-svc` 只提到 "通知/消息推送"，没有具体实现方案。

**需要实时通知的场景**：
- 作业批改完成 → 学生收到通知
- 作品被点赞/评论 → 作者收到通知
- 竞赛开始/结束 → 参与者收到通知
- 班级宠物升级 → 全班收到通知
- 作业截止提醒 → 学生收到通知

**建议**：

```
WebSocket 连接管理 (复用教学云变量的 WebSocket)
  ├── 心跳保活
  ├── 消息推送 (作业/评论/点赞/系统)
  ├── 离线消息存储 (用户不在线时存数据库)
  └── 推送渠道:
       ├── 站内 WebSocket (在线时)
       ├── 浏览器推送 (离线时)
       └── 微信模板消息 (可选)
```

### 问题 6：内容审核流程缺失

**现状**：只提到 "AI 内容审核"，没有具体流程。

**市面参照**：编程猫有完整的审核流程：
- 作品发布 → AI 初筛（敏感词/图片）→ 人工复审队列 → 通过/拒绝
- 评论发布 → 实时敏感词过滤 → 异常内容人工审核

**建议**：

```
内容审核流水线:
  用户提交内容 (作品/评论)
    → Step 1: 规则引擎 (敏感词/正则/黑名单) — < 10ms
    → Step 2: AI 审核 (语义分析/图片识别) — < 3s
    → Step 3: 评分 > 0.8 直接通过, < 0.3 直接拒绝, 中间进人工队列
    → Step 4: 人工审核 (教师/管理员)
    → 通过 → 发布
    → 拒绝 → 通知用户 + 原因
```

---

### 问题 7：缺少搜索架构

**现状**：提到 ES 用于 "搜索/统计"，但没有具体设计。

**搜索场景**：
- 作品搜索（关键词 + 标签 + 类型 + 难度）
- 题目搜索（知识点 + 难度 + 题型）
- 用户搜索
- 学情报表查询

**建议**：

```
ES 索引设计:
  ├── works_index: 作品 (标题/描述/标签/作者/点赞数)
  ├── problems_index: 题目 (标题/知识点/难度/AC率)
  ├── users_index: 用户 (昵称/简介)
  └── analytics_index: 学情数据 (按时间/班级/学生聚合)
```

---

### 问题 8：班级宠物系统的防刷机制

**现状**：积分获取方式列举了，但没有防刷设计。

**风险**：
- 学生用脚本自动登录刷每日积分
- 用小号给自己的作品点赞刷积分
- 重复提交同一个作品刷提交积分

**建议**：

```
防刷机制:
  ├── 每日积分上限 (个人: 200/天)
  ├── 同一作品重复提交不计分
  ├── 同一用户对同一作品只能点赞 1 次
  ├── 新注册用户 24 小时内积分减半
  ├── 异常检测: 短时间大量操作 → 触发验证码
  └── 班级宠物: 积分来源可追溯，异常积分可回滚
```

## 🟢 优化建议

### 建议 1：增加 CDN 和静态资源架构

Scratch 项目包含大量图片/声音素材，CDN 是必须的。

```
静态资源架构:
  ├── 素材文件 (角色/背景/声音) → MinIO → CDN
  ├── sb3 文件 → MinIO → CDN
  ├── 用户头像/封面 → MinIO → CDN
  ├── 前端静态文件 → Nginx → CDN
  └── Scratch 编辑器资源 (scratch-vm/gui) → CDN
```

### 建议 2：增加数据埋点和分析流水线

学情统计需要原始数据支撑。

```
数据采集:
  ├── 前端埋点 (页面浏览/按钮点击/停留时长)
  ├── 后端埋点 (API 调用/判题结果/积分变动)
  └── 数据流向:
       埋点 → Kafka → Flink/ClickHouse → 分析报表
```

### 建议 3：Scratch 编辑器的加载策略

Scratch 编辑器是重资源应用（scratch-vm + scratch-gui 合计 ~5MB），需要优化加载。

```
加载策略:
  ├── 按需加载: 首页不加载编辑器，进入创作页才加载
  ├── 预加载: 用户可能要创作时提前加载
  ├── Service Worker 缓存: 编辑器资源离线可用
  └── 版本管理: scratch-vm 版本与后端解析器版本保持一致
```

### 建议 4：多租户架构预埋

如果后续要卖给机构，需要多租户隔离。

```
多租户方案 (数据隔离):
  ├── 方案 A: 共享数据库 + tenant_id 字段 (简单，适合中小机构)
  ├── 方案 B: 每机构独立 Schema (中等隔离)
  └── 方案 C: 每机构独立数据库 (强隔离，大客户)

当前建议: 预埋 tenant_id 字段，但不实现多租户逻辑
  → 后续需要时只需加 WHERE tenant_id = ? 即可
```

## 修正后的推荐架构

```
┌─────────────────────────────────────────────────┐
│                   前端 (Vue 3)                    │
│  编辑器页 | 社区页 | 刷题页 | 教学页 | 管理后台    │
└────────────────────┬────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────┐
│              Nginx (反向代理 + 静态资源 + CDN)     │
├─────────────────────────────────────────────────┤
│              模块化单体 (Spring Boot 3)            │
│  ┌────────┬────────┬────────┬────────┬────────┐ │
│  │ user   │ editor │ social │ problem│homework│ │
│  │ 模块   │ 模块   │ 模块   │+judge  │ 模块   │ │
│  ├────────┼────────┼────────┼────────┼────────┤ │
│  │ points │analytics│ file  │  msg   │ sb3    │ │
│  │ 模块   │ 模块   │ 模块   │ 模块   │ parser │ │
│  └────────┴────────┴────────┴────────┴────────┘ │
├─────────────────────────────────────────────────┤
│  独立服务 1: 判题沙箱 (Docker 容器池)              │
│  独立服务 2: AI Gateway (LLM 调用/缓存/降级)      │
├─────────────────────────────────────────────────┤
│  MySQL | MongoDB | Redis | MinIO | ES            │
└─────────────────────────────────────────────────┘
```

**服务数量**：从 13 个 → **1 个单体 + 2 个独立服务 = 3 个部署单元**

**好处**：
- 开发快：一个项目搞定
- 部署简：3 个 Docker 容器
- 调试易：本地一个进程跑起来
- 扩展灵活：真正有性能瓶颈时再拆

```
    ↑ 被调用
    ├── editor 模块 (保存作品时解析)
    ├── social 模块 (发布时提取信息)

    ↑ 被调用
    ├── problem 模块 (OJ 刷题场景)

ai-gateway (独立服务)
    ↑ 被调用
    ├── social 模块 (AI 点评)
    ├── problem 模块 (AI 判题/出题)
    ├── homework 模块 (AI 批改)
    └── analytics 模块 (AI 学情分析)
```

## 总结

| 维度 | 当前设计 | 修正后 |
|---|---|---|
| 架构风格 | 13 个微服务 | 1 个模块化单体 + 2 个独立服务 |
| sb3 解析 | 独立微服务 | 共享库 (SDK) |
| 判题引擎 | 独立服务 | 独立服务 (沙箱需要隔离) |
| AI 服务 | 模糊的 "调用外部 LLM" | AI Gateway (Prompt管理/缓存/降级/成本控制) |
| OJ vs 作业 | 两个独立服务 | 统一 judge-core + 不同业务包装 |
| 内容审核 | 缺失 | 三级审核流水线 |
| 实时通知 | 缺失 | WebSocket 推送 |
| 搜索 | 提到 ES 但无设计 | 具体索引设计 |
| 防刷 | 缺失 | 多层防刷机制 |
| CDN | 缺失 | 静态资源 CDN 方案 |
| 数据埋点 | 缺失 | 埋点 → 分析流水线 |
| 多租户 | 未考虑 | 预埋 tenant_id |
