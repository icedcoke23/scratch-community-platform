## ADR-001: 选择模块化单体而非微服务

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 项目初期 |
| **决策者** | 核心开发团队 |

### 上下文

项目初期，团队规模较小（3-5 人），需要快速迭代上线 MVP 版本。同时面临以下选择：

- **微服务架构**：每个模块独立部署，独立数据库，服务间通过 API 通信
- **模块化单体**：单个 Spring Boot 应用，内部按模块划分包结构
- **传统单体**：不分模块，所有代码混在一起

### 决策

使用 **Spring Boot 模块化单体**架构，项目划分为 14 个 Maven 模块：

```
backend/
├── scratch-app            # 启动模块 + 调度器 + 健康检查
├── scratch-common-core    # 公共核心层 (返回体/异常/事件/工具/配置)
├── scratch-common-redis   # 公共 Redis 层 (分布式锁/限流/缓存/调度锁)
├── scratch-common-security# 公共安全层 (JWT/认证/角色/幂等性)
├── scratch-common-audit   # 公共审核层 (敏感词过滤)
├── scratch-sb3            # sb3 解析库 (共享库)
├── scratch-judge-core     # 判题核心库 (共享库)
├── scratch-user           # 用户系统
├── scratch-editor         # 创作引擎
├── scratch-social         # 社区系统
├── scratch-judge          # 判题系统
├── scratch-classroom      # 教室管理
└── scratch-system         # 系统管理
```

### 理由

1. **减少运维复杂度**：单个 JAR 包部署，无需服务注册发现、链路追踪等基础设施
2. **保持开发速度**：模块间方法调用替代 RPC，调试方便
3. **模块边界清晰**：Maven 模块强制定义依赖方向，避免循环依赖
4. **事务一致性**：本地事务保证跨模块操作的 ACID

### 后果

- ✅ 开发效率高，适合小团队快速迭代
- ✅ 部署简单，运维成本低
- ⚠️ 未来可按模块拆分为微服务（模块边界已定义清晰）
- ⚠️ 单机性能有上限，需关注 JVM 内存和线程池配置

## ADR-002: MyBatis-Plus 而非 JPA

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 项目初期 |
| **决策者** | 核心开发团队 |

### 上下文

项目需要选择 ORM 框架，主要候选方案：

- **Spring Data JPA / Hibernate**：全自动 ORM，面向对象映射
- **MyBatis-Plus**：半自动 ORM，SQL 可控
- **原生 MyBatis**：纯手动 SQL 映射

### 决策

使用 **MyBatis-Plus 3.5** 作为 ORM 框架。

### 理由

1. **灵活的 SQL 控制**：复杂查询可直接编写 SQL，不受 ORM 约束
2. **Lambda 查询**：类型安全的条件构建器，避免字段名硬编码
3. **自动填充**：自动维护 `created_at`、`updated_at` 等审计字段
4. **逻辑删除**：内置 `@TableLogic` 注解，一行配置实现软删除
5. **分页插件**：内置分页功能，无需额外集成
6. **批量操作**：支持批量插入、更新，性能优于 JPA

### 后果

- ✅ 开发效率高，CRUD 零 SQL
- ✅ 复杂查询灵活可控
- ✅ 团队成员 MyBatis 经验丰富，学习成本低
- ⚠️ 需要手动维护 Mapper XML 文件
- ⚠️ 实体类与数据库字段需要手动保持同步

## ADR-003: Node.js 判题沙箱

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 判题模块开发阶段 |
| **决策者** | 判题模块负责人 |

### 上下文

Scratch 3.0 项目基于 `scratch-vm`（Node.js 库）运行，需要设计判题执行环境。可选方案：

- **Node.js 子进程**：Java 后端通过 `ProcessBuilder` 启动 Node.js 子进程
- **Docker 容器**：每次判题启动一个 Docker 容器
- **浏览器沙箱**：使用 Puppeteer 无头浏览器运行 Scratch 项目

### 决策

使用 **Node.js 子进程隔离运行 scratch-vm**，通过 Java 的 `ProcessBuilder` 管理。

### 理由

1. **原生支持**：scratch-vm 是 Node.js 库，无需额外适配
2. **启动速度快**：子进程启动开销远小于 Docker 容器
3. **资源可控**：可通过 `ulimit` 和超时机制限制资源使用
4. **实现简单**：无需引入容器编排基础设施

### 后果

- ✅ 判题性能好，启动延迟低
- ✅ 实现复杂度适中
- ⚠️ 需要额外的进程管理（超时、僵尸进程清理）
- ⚠️ 无 OS 级隔离，安全性弱于 Docker（待后续优化）

## ADR-004: TurboWarp iframe 嵌入编辑器

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 前端编辑器集成阶段 |
| **决策者** | 前端负责人 |

### 上下文

平台需要集成 Scratch 编辑器，让用户在线创建和编辑项目。可选方案：

- **Fork scratch-gui**：克隆 scratch-gui 源码，深度定制
- **TurboWarp iframe 嵌入**：使用 TurboWarp 提供的嵌入式编辑器
- **自研编辑器**：从零开发积木编辑器

### 决策

使用 **TurboWarp iframe 嵌入**方式集成编辑器。

### 理由

1. **技术栈兼容**：scratch-gui 是 React 应用，与项目的 Vue 技术栈不兼容
2. **维护成本低**：无需维护 scratch-gui 的 fork，TurboWarp 团队持续更新
3. **功能增强**：TurboWarp 相比原生 Scratch 有更好的性能和更多功能
4. **开发速度快**：iframe 嵌入 + postMessage 通信，数天即可集成

### 后果

- ✅ 开发速度快，维护成本低
- ✅ 编辑器功能丰富，性能优于原生 Scratch
- ⚠️ 依赖第三方服务（TurboWarp），需关注可用性
- ⚠️ 通信受限于 postMessage API，深度定制能力有限
- ⚠️ 需要处理跨域通信的安全问题

## ADR-005: DFA 敏感词过滤

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 内容安全模块开发阶段 |
| **决策者** | 安全模块负责人 |

### 上下文

作为少儿编程平台，用户（主要是儿童）发布的内容需要进行敏感词过滤。可选方案：

- **正则表达式匹配**：为每个敏感词编写正则
- **DFA（确定性有限自动机）**：将敏感词构建成状态机
- **第三方内容审核 API**：调用阿里云/腾讯云内容审核服务
- **AC 自动机**：多模式匹配算法

### 决策

使用 **DFA（确定性有限自动机）** 实现本地敏感词过滤。

### 理由

1. **时间复杂度 O(n)**：单次遍历文本即可完成匹配，性能优异
2. **内存占用可控**：状态机内存占用与敏感词数量线性相关
3. **无外部依赖**：纯本地运行，无需网络调用，无延迟
4. **可离线使用**：不依赖第三方服务可用性

### 后果

- ✅ 过滤性能好，延迟低
- ✅ 无外部依赖，稳定性高
- ⚠️ 需要维护敏感词库（定期更新）
- ⚠️ 无法识别变体写法（谐音、拼音等，需额外处理）

## ADR-006: Spring Event 事件驱动解耦

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已采纳 |
| **日期** | 社区模块开发阶段 |
| **决策者** | 后端架构师 |

### 上下文

跨模块写操作（如用户点赞项目 → 更新项目点赞数、更新用户积分、发送通知）需要解耦。可选方案：

- **直接调用**：在 Service 层直接注入其他模块的 Service
- **Spring Event**：使用 Spring 内置事件机制
- **消息队列**：引入 RabbitMQ / Kafka
- **观察者模式**：手动实现发布-订阅

### 决策

使用 **Spring Event + @TransactionalEventListener** 实现跨模块解耦。

### 理由

1. **事务一致性**：`@TransactionalEventListener` 保证事件在事务提交后才处理
2. **模块解耦**：事件发布者不需要知道有哪些监听者
3. **无外部依赖**：Spring 内置机制，无需引入消息队列
4. **开发简单**：基于注解，代码量少，易于维护

### 后果

- ✅ 模块间耦合度低，易于扩展
- ✅ 事务一致性有保障
- ⚠️ 需要处理事件发布失败的降级逻辑
- ⚠️ 不支持跨 JVM 的分布式事件（未来需要时可迁移到消息队列）
