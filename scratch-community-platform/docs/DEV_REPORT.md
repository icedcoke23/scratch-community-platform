# Scratch 社区平台开发报告

## 项目概述

Scratch 社区平台是一个基于 TurboWarp 的自托管在线编程教育平台，旨在为用户提供完整的 Scratch 项目创建、编辑、预览和分享功能。平台采用现代化的前后端分离架构，前端使用 Vue 3 构建，后端使用 Express.js 实现，TurboWarp 编辑器通过创新的 wrapper iframe 方案进行集成。本报告记录了平台的开发历程、技术决策、已完成的各项功能以及后续的迭代计划。

## 开发历程

### 第一阶段：项目初始化与基础架构搭建

项目的初始阶段主要完成了技术选型和基础架构的搭建工作。在充分调研了现有的 Scratch 在线编辑器方案后，团队决定采用 TurboWarp 作为核心编辑器引擎。TurboWarp 是一个对 Scratch 进行优化的修改版本，提供了更快的执行速度和更丰富的功能，同时保持了与 Scratch 的完全兼容性。

前端技术栈选择了 Vue 3 作为核心框架，主要考虑到其优秀的响应式系统、Composition API 的灵活性以及活跃的社区生态。构建工具选择了 Vite 5，这是一个基于 ES Module 的新一代构建工具，相比传统的 Webpack 具有更快的冷启动速度和更高效的生产构建能力。UI 组件库选用了 Element Plus，它为 Vue 3 提供了完善的桌面端组件支持。

后端技术栈选择了 Express.js 作为 Web 框架，这是 Node.js 生态中最成熟稳定的框架之一，具有简洁的 API 和丰富的中间件生态。TypeScript 被选为主要编程语言，通过静态类型检查提高代码质量和可维护性。数据存储方面，考虑到项目定位于社区平台，初期采用内存存储简化部署和测试，后续可以根据需求迁移到专业的数据库系统。

### 第二阶段：TurboWarp 集成方案设计与实现

TurboWarp 集成是本项目最具挑战性的技术难点之一。TurboWarp 官方提供了网页版编辑器，但直接集成到第三方平台存在跨域通信的挑战。经过深入分析和多轮方案讨论，最终采用了本地 wrapper iframe 方案来解决这个问题。

方案的核心思路是在本地托管 editor.html 和 player.html 作为 wrapper 页面，这些 wrapper 页面通过 iframe 加载 TurboWarp 官方 CDN 的内容。由于 wrapper 页面与前端应用同源，postMessage 通信可以在安全受控的环境中进行。wrapper 页面负责转发消息，使得前端应用可以间接控制 TurboWarp 的行为。

为了封装 postMessage 通信的复杂性，开发了 ScratchBridge 模块。该模块提供了类型安全的接口，定义了所有消息类型的 TypeScript 接口，支持灵活的事件回调机制。开发者可以通过简洁的 API 控制编辑器播放、暂停、停止等操作，监听编辑器状态变化和项目变更事件。ScratchBridge 模块的设计遵循了桥接模式的思想，将复杂的通信细节封装在模块内部，对外提供清晰的接口。

TurboWarp 配置管理模块负责管理编辑器相关的 URL 配置，支持在开发和生产环境之间灵活切换。开发环境下可以使用远程 TurboWarp 加速调试，生产环境使用本地构建的版本确保稳定性和可控性。配置模块还提供了 URL 构建工具函数，支持自定义主题、帧率、编译器选项等参数。

### 第三阶段：功能完善与优化

在核心功能稳定后，团队进行了多轮功能完善和性能优化。前端方面，完善了项目管理界面，包括项目列表页、创建项目表单、编辑项目详情等。预览组件增加了全屏播放支持，改进了加载状态的展示。编辑器页面整合了项目管理工具栏，支持保存项目到后端服务器。

后端方面，实现了完整的项目管理 API，包括项目的增删改查操作、文件上传下载功能。项目 API 遵循 RESTful 设计规范，响应格式统一包含状态码、数据和消息字段，便于前端进行处理。中间件层实现了请求日志、错误处理、数据验证等横切关注点。

性能优化方面，Nginx 配置了静态资源缓存策略，TurboWarp 相关的 JS/CSS 文件设置了 30 天的缓存时间。Gzip 压缩启用于减少传输数据量。前端构建产物进行了代码分割和懒加载，减少首屏加载时间。

## 解决的问题与技术决策

### 跨域通信问题

项目面临的首要技术挑战是前端应用与 TurboWarp 编辑器之间的跨域通信问题。TurboWarp 官方编辑器部署在 turbowarp.org 域名下，而平台的前端应用可能部署在不同的域名或端口上。浏览器的同源策略限制了跨域的 postMessage 通信，直接集成会遇到安全限制。

为了解决这个问题，团队评估了三种可能的方案。方案一是使用 postMessage API 直接与远程 TurboWarp 通信，但这种方法需要正确配置 CORS 头，且安全风险较高。方案二是完全自托管 TurboWarp，即下载并部署 TurboWarp 的所有依赖资源到本地服务器，这种方法可以完全控制资源但部署复杂且更新困难。方案三是采用本地 wrapper iframe 方案，即在本地托管简化的 HTML 页面作为 wrapper，通过这些 wrapper 页面转发消息。

最终选择方案三是因为它在安全性和维护性之间取得了最佳平衡。Wrapper 页面与前端应用同源，postMessage 通信不会受到跨域限制。同时，wrapper 页面的职责单一明确，只负责转发消息，不涉及复杂的业务逻辑，更新 TurboWarp 时只需重新构建即可。

### postMessage 通信协议设计

消息通信协议的设计需要考虑多个方面：消息类型的安全性、参数传递的可靠性、以及接口的易用性。最初的设计尝试实现完整的双向协议，包括导出项目、切换模式等复杂功能。但在实际开发中发现，过度复杂的协议设计反而增加了调试难度和出错概率。

经过迭代优化，协议简化为两个核心功能：项目加载和播放控制。项目加载通过 URL 参数直接传递项目地址，TurboWarp 会自动处理项目文件的获取和加载。播放控制通过简单的消息类型（play、stop、pause）实现，TurboWarp 端负责处理具体的播放逻辑。

协议采用消息类型驱动的设计，所有消息都包含 `type` 字段用于标识消息类型。这种设计模式使得添加新的消息类型变得简单，只需在消息处理器中添加对应的 case 分支即可。TypeScript 的类型定义确保了消息格式的一致性，编译时就能发现类型错误。

### 架构分层与模块组织

前后端都采用了分层架构设计，明确各层的职责边界。前端分为表现层（Vue 组件）、业务逻辑层（Composables/Stores）、通信层（API 模块和 ScratchBridge）。后端分为控制器层（处理请求响应）、服务层（实现业务逻辑）、数据访问层（内存存储）。

分层架构的优势在于关注点分离和代码复用。每个层只需关注自己的职责，不需要了解其他层的实现细节。同时，相同的业务逻辑可以在不同的地方复用，比如项目验证逻辑既可以在创建项目时使用，也可以在更新项目时使用。

模块组织方面，前端按照功能类型而非组件类型组织代码。相关的 API 调用、工具函数、类型定义都放在同一个目录下，而不是按照组件分散在不同位置。这种组织方式使得查找和维护相关代码更加方便。

## 架构变更记录

### v1.0.0 基础架构版本

初始架构采用单体设计，前端和后端代码在同一个仓库中，通过 Nginx 反向代理协调。这种设计简化了部署，但随着功能增加，代码量也在快速增长。路由配置通过 Vue Router 的动态导入实现懒加载，减小了首屏 bundle 体积。

### v1.1.0 TurboWarp 集成版本

TurboWarp 集成是架构变更最大的版本。新增了 ScratchBridge 模块作为前端与编辑器通信的桥梁。TurboWarp 的构建产物单独存放在 `frontend-vue/public/turbowarp/` 目录，与前端源码分离。这种分离使得 TurboWarp 的更新可以独立进行，不需要重新构建整个前端应用。

### v1.2.0 配置管理版本

新增了 turbowarpConfig 模块统一管理 TurboWarp 相关配置。配置分为开发环境和生产环境两套，开发环境使用远程 TurboWarp CDN，生产环境使用本地构建版本。配置模块还提供了 URL 构建函数，封装了复杂的 URL 参数拼接逻辑。

### v1.3.0 API 完善版本

后端 API 进行了重构，从简单的路由处理函数分离出控制器、服务层。引入了中间件机制处理横切关注点，如日志记录、错误处理、数据验证。响应格式进行了标准化统一，包含 code、data、msg、timestamp 四个字段。

## 已知问题与限制

### 功能限制

内存存储不支持数据持久化，服务重启后所有数据会丢失。这对于生产环境是不可接受的，后续需要迁移到支持持久化的数据库系统。当前仅支持基本的项目管理和文件操作，缺少用户认证和权限管理功能。TurboWarp 编辑器的自定义扩展加载功能尚未实现。

### 性能限制

TurboWarp 首次加载需要下载大量 JavaScript 和 CSS 资源，可能导致首次访问速度较慢。当前架构不支持多服务器水平扩展，所有请求都路由到单个后端实例。文件上传大小限制为 50MB，更大的 Scratch 项目可能无法上传。

### 兼容性限制

平台主要针对桌面浏览器进行优化，移动端的用户体验可能不如桌面端。部分浏览器可能因安全策略差异导致功能异常。TurboWarp 编辑器依赖现代浏览器特性，不支持 Internet Explorer 等旧版浏览器。

## 下一步开发计划

### 短期计划（1-3个月）

短期计划主要集中在功能完善和稳定性提升方面。首先需要完成数据持久化改造，将内存存储迁移到 PostgreSQL 或 MongoDB 等数据库系统。选择 PostgreSQL 是因为其强大的关系数据管理能力和事务支持，适合管理项目之间的关联关系。数据库迁移需要设计合理的数据模型，包括用户表、项目表、文件表等核心实体。

其次是实现用户认证和权限管理功能。计划采用 JWT（JSON Web Token）进行身份认证，支持用户注册、登录、密码找回等基本功能。权限管理需要区分普通用户和管理员，普通用户只能操作自己的项目，管理员可以管理所有项目。

最后是完善错误处理和用户提示。当前的错误处理比较简单，用户可能看到不够友好的错误信息。需要设计统一的错误页面和提示机制，引导用户正确使用平台功能。

### 中期计划（3-6个月）

中期计划聚焦于平台扩展和性能优化。考虑引入 Redis 缓存层，提高频繁访问数据的读取速度。实现后端服务的多实例部署，通过负载均衡分散请求压力。添加实时通信功能，支持用户之间的在线协作编辑。

探索 TurboWarp 扩展生态，支持加载第三方 Scratch 扩展。这需要研究 Scratch 扩展的加载机制和沙箱安全策略。考虑开发平台专属的扩展功能，如云变量、物联网设备支持等。

优化前端性能，包括资源预加载、代码分割细化、图片懒加载等。考虑引入 PWA（Progressive Web App）技术，支持离线访问和原生应用体验。

### 长期计划（6-12个月）

长期计划着眼于平台生态和商业化探索。建立开发者 API，允许第三方应用接入平台功能。考虑引入项目分享和社区互动功能，如评论、点赞、关注等。探索教育场景的深度整合，如班级管理、作业布置、学习分析等功能。

安全性方面需要进行全面的安全审计，包括渗透测试、依赖包漏洞扫描等。性能方面可以考虑引入 CDN 加速全球访问，使用对象存储服务管理项目文件。考虑通过 Docker 和 Kubernetes 实现容器化部署和自动扩缩容。

## 技术债务与改进建议

### 代码质量

当前代码库中部分模块的单元测试覆盖不足，特别是后端业务逻辑层。建议引入 Jest 或 Mocha 框架编写单元测试和集成测试，确保核心功能的正确性。前端组件的测试可以使用 Vue Test Utils 进行组件级别的测试。

代码风格方面，部分文件的注释和文档不够完善。建议引入 JSDoc 规范编写 API 文档，使用 TypeScript 的严格模式提高类型安全。考虑引入预提交钩子自动检查代码格式和类型错误。

### 部署运维

当前的部署流程需要手动执行多个步骤，存在操作失误的风险。建议编写自动化部署脚本或使用 CI/CD 流水线实现一键部署。日志收集和分析机制需要完善，便于问题排查和性能监控。

容器化部署方案需要进一步优化，包括多阶段构建减少镜像体积、资源限制配置、health check 实现等。考虑引入 Kubernetes 实现生产环境的容器编排和高可用部署。

### 性能监控

缺乏完善的性能监控和告警机制。建议引入 APM（Application Performance Monitoring）工具监控应用的响应时间、错误率等关键指标。配置日志告警规则，及时发现和处理异常情况。

## 总结

Scratch 社区平台经过多个阶段的开发，已经完成了基础架构搭建、TurboWarp 集成和核心功能实现。平台采用了现代化的前后端分离架构，技术栈选择合理，代码组织清晰。TurboWarp 的 wrapper iframe 集成方案是一个创新的解决方案，在安全性和易用性之间取得了平衡。

当前平台已经具备了基本的使用价值，可以用于 Scratch 项目的在线创建和编辑。后续需要重点解决数据持久化、用户认证等生产环境必需的功能，同时不断完善文档和优化用户体验。团队需要持续关注技术发展趋势，及时引入新技术提升平台竞争力。

通过本项目的开发实践，团队积累了 Vue 3、Express.js、TypeScript 等技术的实战经验，掌握了前后端分离架构的设计方法，培养了解决复杂技术问题的能力。这些经验和能力将为后续的项目开发提供有力的支持。
