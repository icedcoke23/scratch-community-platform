# Scratch 社区平台架构设计文档

本文档详细描述了 Scratch 社区平台的技术架构设计，包括系统架构、前后端设计、数据存储、API 设计以及安全考虑等各个方面。通过阅读本文档，开发者和运维人员可以全面了解平台的技术实现细节，为后续的开发和部署工作提供参考依据。

## 系统架构概述

### 整体架构设计

Scratch 社区平台采用了经典的前后端分离架构设计，整个系统由前端应用、后端服务和辅助工具三个主要部分组成。前端基于 Vue 3 框架构建，采用单页面应用（SPA）的模式运行，为用户提供交互友好的 Scratch 项目编辑和预览界面。后端基于 Express.js 框架构建，提供 RESTful 风格的 API 服务，负责处理业务逻辑和数据持久化工作。TurboWarp 编辑器作为核心功能组件，通过 iframe 方式集成到前端应用中，实现了完整的在线编程环境。

平台的技术选型遵循了现代化、轻量化和可维护的原则。前端使用 Vue 3 的 Composition API 进行开发，充分利用了其响应式系统的优势，使得状态管理和组件逻辑更加清晰。后端使用 TypeScript 进行开发，通过静态类型检查和面向接口的编程方式，提高了代码的可靠性和可维护性。整体架构采用了分层设计的思想，将表现层、业务逻辑层和数据访问层分离，降低了各层之间的耦合度，便于后续的功能扩展和维护升级。

### 架构分层说明

整个平台的技术架构可以划分为以下几个层次，每个层次都有其明确的职责边界和技术特点。了解这些层次的划分有助于开发者在开发过程中准确定位代码的位置，也有助于运维人员在故障排查时快速定位问题所在。

表现层是用户直接交互的界面部分，由 Vue 3 单页面应用构成。该层负责渲染用户界面、处理用户输入、调用后端 API 以及管理前端路由。表现层使用了 Element Plus 组件库来构建统一的 UI 风格，通过 Pinia 进行前端状态管理，使用 Vue Router 处理页面导航。在表现层中，TurboWarp 编辑器通过 iframe 的方式嵌入，ScratchBridge 模块负责与编辑器进行双向通信。

网关层是所有用户请求的入口点，由 Nginx 服务器承担这一角色。网关层的主要职责包括：提供静态文件服务，包括前端构建产物和 TurboWarp 编辑器资源；将 API 请求反向代理到后端服务；处理 HTTPS 终端，完成 SSL/TLS 加密解密；实施访问控制和请求限流策略；配置缓存策略以提升静态资源访问性能。网关层的配置直接影响到平台的安全性和访问性能，因此需要仔细规划和调优。

业务逻辑层是平台的核心部分，由 Express.js 后端服务实现。该层负责处理所有业务请求，包括项目管理、文件操作、用户认证等核心功能。业务逻辑层采用控制器的设计模式，将请求处理、业务逻辑和数据访问分离。控制器负责接收请求和返回响应，服务层实现具体的业务逻辑，中间件负责处理跨领域关注点，如日志记录、错误处理和请求验证等。

数据层负责数据的持久化和访问，目前平台采用内存存储的方式管理数据。内存存储适用于开发和演示环境，具有读写速度快的优点，但需要注意数据的持久化问题。平台提供了文件上传下载功能，SB3 格式的项目文件通过 Multer 中间件处理，支持大文件上传和格式验证。

## 前端架构设计

### 技术栈概览

前端应用基于 Vue 3 框架构建，采用现代化的前端开发技术栈。选择 Vue 3 作为核心框架是因为其提供了优秀的响应式系统、灵活的组合式 API 以及完善的生态系统。Vue 3 的 Composition API 允许开发者使用函数式的方式组织组件逻辑，使得代码更加模块化和可复用。同时，Vue 3 还保持了与 Vue 2 的兼容性，降低了学习成本和迁移风险。

构建工具选择了 Vite 5，这是一个基于 ES Module 的下一代前端构建工具。相比传统的 Webpack，Vite 在开发环境下具有极速的冷启动速度，因为它利用了浏览器的原生 ES Module 支持，无需预先打包所有模块。在生产环境下，Vite 使用 Rollup 进行优化打包，能够生成体积小、加载快的前端资源。

路由管理使用 Vue Router 4，这是 Vue 官方提供的路由管理器。Vue Router 4 支持路由懒加载、导航守卫、路由元信息等功能，能够满足复杂单页面应用的路由需求。平台的主要页面包括首页、项目列表页、预览演示页和编辑器页面，每条路由都配置了相应的组件和元信息。

状态管理采用 Pinia，这是 Vue 官方推荐的新一代状态管理库。相比 Vuex，Pinia 提供了更简洁的 API 和更好的 TypeScript 支持。Pinia 使用 stores 的概念组织状态，每个 store 是一个独立的响应式状态容器，可以包含 state、getters 和 actions。平台目前使用 user store 管理用户相关的状态信息。

UI 组件库选用了 Element Plus，这是一个为 Vue 3 打造的桌面端组件库。Element Plus 提供了丰富的 UI 组件，包括按钮、表单、表格、对话框等常用组件，能够快速构建一致性良好的用户界面。组件库采用主题定制机制，可以根据平台的设计规范调整样式风格。

HTTP 客户端使用 Axios，这是一个基于 Promise 的 HTTP 请求库。Axios 提供了简洁的 API 和丰富的配置选项，支持请求拦截器、响应拦截器、自动转换 JSON 等功能。平台在 Axios 基础上封装了统一的请求服务，集成了用户认证令牌管理和错误处理逻辑。

### 目录结构与模块划分

前端项目的目录结构遵循了 Vue 生态的最佳实践，按照功能类型而非组件类型组织代码。这种组织方式使得相关功能的代码聚集在一起，便于开发者查找和维护。整个前端项目的目录结构如下所示：

api 目录包含了所有与后端 API 交互的代码。index.ts 文件是 API 服务的统一封装，定义了请求拦截器、响应拦截器和错误处理逻辑。该模块导出了项目相关的 API 方法，包括获取项目列表、获取项目详情、创建项目、更新项目、删除项目和上传下载 SB3 文件等功能。每个 API 方法都封装了请求参数处理和响应数据转换的逻辑，调用方只需关注业务数据的处理。

components 目录存放可复用的 Vue 组件。ScratchPreview.vue 是项目的预览组件，负责在 iframe 中加载 TurboWarp 播放器并提供播放控制功能。该组件封装了与播放器交互的细节，提供了简洁的属性接口和事件接口，便于在各种场景中复用。

router 目录定义了应用的路由配置。每条路由指定了访问路径、对应组件和可选的元信息。平台配置了路由懒加载，只有在访问特定页面时才加载对应的组件代码，这样可以减少首屏加载时间。路由守卫用于处理页面访问权限控制和导航逻辑。

stores 目录包含 Pinia 状态管理 stores。user.ts store 管理用户相关的状态，包括用户信息、登录状态和认证令牌等。状态管理采用响应式设计，当状态发生变化时，依赖该状态的组件会自动重新渲染。

utils 目录包含了工具函数和辅助模块。scratchBridge.ts 实现了与 TurboWarp iframe 通信的核心逻辑，封装了 postMessage 消息的发送和接收。turbowarpConfig.ts 提供了 TurboWarp 的配置管理，包括编辑器 URL、播放器 URL 和环境切换等功能。

views 目录包含页面级组件，每个组件对应一条路由。HomeView 是平台首页，展示平台介绍和使用指引。ProjectListView 是项目列表页面，提供项目浏览、搜索和筛选功能。PreviewDemoView 是预览功能的演示页面。ScratchEditorView 是 Scratch 编辑器页面，集成了 TurboWarp 编辑器并提供了项目管理功能。

### 核心组件设计

ScratchPreview 组件是平台的核心组件之一，负责在网页中嵌入 TurboWarp 播放器并提供交互能力。该组件的设计考虑了易用性和可定制性，提供了丰富的配置选项和事件回调。

组件接收以下属性参数：projectUrl 指定要加载的项目地址，可以是相对路径或绝对路径；autoplay 控制是否自动开始播放；controls 控制播放器控制栏的显示；fullscreen 控制全屏按钮的可用性。组件触发的事件包括：projectLoaded 当项目加载完成时触发；playStateChanged 当播放状态改变时触发；error 当发生错误时触发。

组件内部使用 iframe 加载 TurboWarp 播放器页面，通过 URL 参数传递配置选项。当项目 URL 发生变化时，组件会重新设置 iframe 的 src 属性以加载新项目。播放控制通过 ScratchBridge 模块实现，该模块封装了与播放器的 postMessage 通信协议。

ScratchEditorView 是编辑器页面的主组件，负责协调编辑器界面和项目管理功能。该组件整合了 ScratchPreview 组件用于项目预览，同时提供了项目管理工具栏和状态显示区域。

编辑器页面的主要功能包括：加载指定项目到编辑器中进行编辑；预览当前项目的运行效果；保存项目到后端服务器；切换到其他项目进行编辑。组件使用 Vue Router 的路由参数获取项目 ID，如果未提供 ID 则创建新项目。

ScratchBridge 模块是前端与 TurboWarp 通信的核心模块，封装了 postMessage 通信协议的细节。该模块提供了类型安全的接口和灵活的事件回调机制，是前后端通信的桥梁。

## 后端架构设计

### 技术栈与框架

后端服务基于 Node.js 运行环境，使用 Express.js 框架构建 RESTful API。Express.js 是 Node.js 生态中最流行的 Web 框架，以其简洁的 API 和灵活的中间件机制著称。选择 Express.js 作为后端框架是因为它学习曲线平缓、文档完善、社区活跃，能够满足平台当前的业务需求。

TypeScript 是后端开发的主要编程语言，它为 JavaScript 添加了静态类型检查和面向对象特性。使用 TypeScript 可以显著提高代码质量，在编译阶段就能发现类型错误和潜在的 bug。同时，TypeScript 的类型系统也是最好的文档形式，开发者可以通过类型定义了解接口的使用方式。

Express.js 的中间件机制是框架的核心特性，平台充分利用了这一机制来实现各种功能。请求日志中间件记录每个请求的详细信息，包括请求方法、路径、参数和响应状态等。错误处理中间件统一捕获和处理所有异常，确保错误能够以合适的格式返回给客户端。CORS 中间件处理跨域资源共享问题，允许前端应用跨域访问 API。请求体解析中间件处理 JSON 和表单数据，支持大文件上传。

### 目录结构与模块划分

后端项目采用分层架构设计，将代码按照职责划分为不同的层次。控制层负责处理请求和返回响应，服务层实现业务逻辑，工具层提供辅助功能。这种分层设计使得代码结构清晰，便于维护和测试。

```
backend/
├── src/
│   ├── controllers/            # 控制器层
│   │   └── projectController.ts # 项目控制器
│   ├── middleware/            # 中间件
│   │   ├── errorHandler.ts    # 错误处理中间件
│   │   ├── notFound.ts       # 404 处理中间件
│   │   ├── requestLogger.ts  # 请求日志中间件
│   │   └── validation.ts     # 数据验证中间件
│   ├── routes/                # 路由定义
│   │   └── projectRoutes.ts  # 项目相关路由
│   ├── services/              # 业务逻辑层
│   │   └── projectService.ts # 项目服务
│   ├── types/                 # TypeScript 类型定义
│   │   ├── errors.ts         # 错误类型定义
│   │   ├── index.ts          # 类型导出
│   │   ├── project.ts        # 项目类型定义
│   │   └── user.ts           # 用户类型定义
│   ├── utils/                # 工具函数
│   │   └── response.ts       # 响应工具
│   ├── app.ts               # 应用主文件
│   └── index.ts             # 服务入口
├── public/                   # 静态文件目录
├── package.json             # 项目配置
└── tsconfig.json            # TypeScript 配置
```

controllers 目录包含控制器文件，每个控制器负责处理一类资源的请求。projectController.ts 是项目相关的控制器，定义了获取项目列表、获取项目详情、创建项目、更新项目、删除项目以及文件上传下载等处理函数。控制器函数接收 Express 的 Request 和 Response 对象，从请求中提取参数，调用服务层完成业务逻辑，然后将结果写入响应。

middleware 目录包含 Express 中间件文件。errorHandler.ts 定义了全局错误处理中间件，捕获所有未处理的异常并返回格式化的错误响应。notFound.ts 处理404错误，当请求的路由不存在时返回友好的提示。requestLogger.ts 记录每个请求的详细信息，用于调试和监控。validation.ts 提供请求数据的验证功能，包括参数类型检查、必填项检查和格式验证。

routes 目录定义了 API 路由。projectRoutes.ts 聚合了项目相关的所有路由，使用 Express Router 将路由分组管理。路由定义包括请求方法（GET、POST、PUT、DELETE）、路由路径和处理函数。路由处理函数通常是控制器方法，也可以直接调用服务层方法。

services 目录包含业务逻辑层的代码。projectService.ts 实现了项目管理的核心业务逻辑，包括项目数据的增删改查操作、文件上传下载处理以及数据验证规则。服务层的代码不直接处理 HTTP 请求，只关注业务逻辑的实现，这使得业务逻辑可以被不同的入口调用，也便于编写单元测试。

types 目录包含 TypeScript 类型定义文件。errors.ts 定义了应用中使用的自定义错误类型。project.ts 定义了项目相关的数据结构，包括项目信息、查询参数和响应格式。类型定义是代码文档的重要组成部分，通过阅读类型定义可以快速了解数据结构和接口契约。

utils 目录包含工具函数。response.ts 提供了统一的响应格式封装，确保所有 API 响应都遵循一致的格式规范。统一的响应格式便于前端进行数据处理，也使得 API 文档更加清晰。

### API 设计规范

平台的 API 设计遵循 RESTful 风格，使用 HTTP 方法表达操作语义，URL 路径表示资源层级关系。所有 API 响应都采用统一的 JSON 格式，包含状态码、数据和消息三个字段。

API 的 URL 结构以 `/api/v1` 为前缀，表示 API 的版本号。版本号机制允许在不破坏现有客户端的情况下进行 API 升级。资源名称使用复数形式，如 `/projects` 表示项目集合，`/projects/:id` 表示单个项目。

响应格式采用标准化的结构：

```json
{
  "code": 0,
  "data": {},
  "msg": "success",
  "timestamp": 1710000000000
}
```

其中 code 字段表示操作结果状态，0 表示成功，非 0 表示失败。data 字段包含业务数据，msg 字段包含状态消息，timestamp 字段表示服务器时间戳。这种响应格式与前端 API 封装模块的类型定义相对应，便于进行类型检查和数据处理。

## 数据存储设计

### 内存存储方案

当前平台采用内存存储方案管理数据，所有数据都存储在 Node.js 进程的内存中。这种方案适用于开发测试和小规模演示，具有实现简单、读写速度快的优点。然而，内存存储也存在明显的局限性：数据不会持久化保存，服务重启后数据会丢失；受限于进程可用内存，不适合大规模数据存储；无法支持多进程部署时的数据共享。

内存存储使用 JavaScript 的 Map 和 Array 等数据结构保存数据。项目数据存储在一个 Map 中，以项目 ID 为键，项目对象为值。这种结构提供了常数时间复杂度的查找性能，能够快速定位指定的项目。

```typescript
// 内存存储数据结构
interface Project {
  id: number;
  title: string;
  description?: string;
  tags?: string;
  status: 'draft' | 'published';
  blockCount?: number;
  complexityScore?: number;
  createdAt: string;
  updatedAt: string;
  sb3Data?: Buffer;
  author?: {
    id: number;
    username: string;
    avatar?: string;
  };
  likeCount: number;
  viewCount: number;
}

// 存储实例
const projects = new Map<number, Project>();
const projectIdCounter = 1;
```

为了模拟真实的数据库行为，内存存储模块提供了类似数据库的增删改查接口。这些接口封装了具体的存储操作细节，上层代码无需关心数据是存储在内存还是数据库中。这种设计使得未来切换到真正的数据库时，只需要修改存储模块的实现，而不影响上层代码。

### 数据操作接口

存储模块提供了以下数据操作接口，每个接口都遵循统一的调用模式和返回格式。

查找所有项目接口返回项目列表，支持分页、筛选和排序。分页参数包括页码和每页数量，筛选参数包括状态和关键词，排序参数包括排序字段和排序方向。接口返回满足条件的所有项目，以及符合条件的项目总数。

根据 ID 查找项目接口返回指定 ID 的项目详情。如果项目不存在，返回 null 或抛出异常。详情接口还会增加项目的浏览计数，记录项目被查看的次数。

创建项目接口生成新的项目记录，分配唯一的 ID，设置创建时间和更新时间。创建时需要提供项目的基本信息，如标题、描述和标签。项目创建后默认为草稿状态，可以随时发布。

更新项目接口修改指定项目的属性，可以更新标题、描述、标签和状态。更新操作会修改 updatedAt 时间戳，记录最后修改时间。更新时需要验证项目是否存在以及当前用户是否有权限修改。

删除项目接口移除指定的项目记录。删除操作是不可逆的，通常需要确认操作以防止误删。删除项目后，相关的 SB3 文件数据也会被清理。

上传 SB3 文件接口保存项目文件数据。SB3 文件是 Scratch 项目的压缩包格式，包含项目中的角色、背景、代码块等信息。文件上传时会验证文件格式和大小，确保上传的是有效的 Scratch 项目文件。

下载 SB3 文件接口返回项目的文件数据。如果项目存在且包含文件数据，返回文件内容供 TurboWarp 编辑器加载。项目文件通过流式传输返回，支持大文件下载。

## 安全考虑

### 跨域资源共享（CORS）

由于平台采用前后端分离架构，前端应用和后端服务可能运行在不同的域名或端口上。浏览器出于安全考虑，默认禁止跨域的 AJAX 请求。CORS（Cross-Origin Resource Sharing）是解决跨域访问的标准方案，通过在响应头中添加特定的字段，告诉浏览器允许来自指定来源的请求。

平台在后端配置了 CORS 中间件，允许来自前端应用的跨域请求。开发环境下，前端应用通常运行在 localhost 的某个端口上，此时 CORS 配置允许所有 localhost 来源的请求。生产环境下，应该限制为具体的域名来源，防止恶意网站伪造请求。

```typescript
// CORS 中间件配置
app.use(cors({
  origin: process.env.NODE_ENV === 'production' 
    ? 'https://your-domain.com' 
    : '*',
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true
}));
```

CORS 的预检请求（OPTIONS 方法）也需要正确处理。浏览器在发送跨域的 PUT、DELETE 请求前，会先发送预检请求询问服务器是否允许该请求。服务器需要正确响应预检请求，返回允许的方法和头信息。

### 内容安全策略（CSP）

内容安全策略是一种额外的安全层，用于检测并削弱某些特定类型的攻击，如跨站脚本（XSS）和数据注入攻击。CSP 通过 HTTP 响应头告知浏览器，哪些外部资源可以加载和执行，从而防止恶意代码注入。

平台需要在响应头中配置适当的 CSP 规则。对于 TurboWarp 编辑器，需要允许加载来自 TurboWarp CDN 的脚本和样式。同时，应该禁止内联脚本执行，防止 XSS 攻击。

```nginx
# Nginx 配置 CSP
add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://turbowarp.org 'unsafe-inline'; style-src 'self' 'unsafe-inline' https://turbowarp.org; frame-src https://turbowarp.org;";
```

### iframe 安全

TurboWarp 编辑器通过 iframe 方式嵌入页面，iframe 的安全性需要特别关注。恶意的 iframe 可能会被用于点击劫持攻击，诱导用户在不知情的情况下执行操作。

平台通过 sandbox 属性限制 iframe 的权限，只允许脚本执行和同源访问，阻止弹窗、表单提交等可能产生安全风险的操作。同时，使用 X-Frame-Options 响应头防止页面被其他网站以 iframe 方式嵌入。

```html
<!-- iframe sandbox 配置 -->
<iframe
  src="/turbowarp/editor.html"
  sandbox="allow-scripts allow-same-origin allow-popups allow-forms"
  allow="autoplay"
></iframe>
```

### 文件上传安全

SB3 文件上传是平台的核心功能之一，也是潜在的安全风险点。恶意的上传文件可能包含可执行代码，对服务器造成威胁。平台采取了多层防护措施确保上传安全。

文件类型验证确保上传的是 SB3 格式的文件。SB3 文件实际上是 ZIP 压缩包，平台通过文件扩展名和 MIME 类型进行初步验证。进一步地，可以在解压后检查文件结构是否符合 Scratch 项目格式。

文件大小限制防止用户上传过大的文件占用服务器资源。平台配置了单文件大小上限为 50MB，这个限制足以容纳大多数 Scratch 项目，同时不会对服务器造成过大负担。

文件名处理避免路径遍历攻击。上传文件的文件名需要经过处理，移除可能导致路径遍历的特殊字符，确保文件被保存到指定的目录中。

### 输入验证

所有来自客户端的数据都需要经过验证才能使用，即使这些数据已经在前端进行了验证。服务端验证是防止恶意请求的最后一道防线。

参数类型检查确保请求参数的类型正确，如 ID 应该是数字、状态应该是枚举值等。必填项检查确保必填参数都提供了值。可选参数需要检查是否提供了有效值。

字符串长度限制防止过长的输入消耗服务器资源或导致数据库问题。内容格式验证确保数据符合预期的格式，如日期格式、邮箱格式等。

## 性能优化

### 前端性能优化

前端应用的加载和运行性能直接影响用户体验。平台采取了多项措施优化前端性能，确保页面能够快速加载和流畅运行。

代码分割和懒加载减少了首屏加载时间。Vue Router 支持路由级别的代码分割，每个页面的代码只在访问时才加载。大型组件如编辑器页面，可以进一步拆分成更小的块，按需加载。

资源压缩和合并减少了传输的数据量。Vite 在生产构建时会使用 Terser 对 JavaScript 代码进行压缩和混淆，使用 cssnano 对 CSS 进行优化。Gzip 压缩可以进一步减小传输体积，Nginx 配置了相应的压缩规则。

静态资源缓存提高了重复访问的速度。TurboWarp 的静态资源设置了较长的缓存时间（30天），因为这些资源的版本更新不频繁。带有哈希值的静态文件名（如 `[name]-[hash].js`）可以在更新时强制浏览器加载新版本。

### 后端性能优化

后端服务的性能优化主要关注响应速度和资源利用效率。平台在后端代码和运行环境两个层面进行优化。

响应压缩减少网络传输时间。Express.js 可以启用响应压缩中间件，对 JSON 响应进行 Gzip 压缩。压缩对于数据量较大的响应效果显著。

请求处理优化提高并发能力。避免在请求处理中进行耗时的同步操作，尽可能使用异步方式处理 I/O。数据库操作应该使用索引优化查询性能。

进程管理优化利用多核 CPU。Node.js 是单线程的，可以通过启动多个进程来利用多核 CPU。PM2 进程管理器可以方便地管理多个 Node.js 进程，实现负载均衡和故障恢复。
