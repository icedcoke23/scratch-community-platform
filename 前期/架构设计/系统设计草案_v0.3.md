# 🏗️ 系统设计草案 v0.3（MVP 聚焦版）

> 日期：2026-04-23 | 基于 v0.2 + 审计报告修正，聚焦 MVP
> 聚焦 Scratch，课后教学场景
> 架构：模块化单体 + 判题沙箱（2 个部署单元）
> 存储：MySQL + Redis + MinIO（与技术选型 v0.3 对齐）

---

## 一、产品定位

**一句话**：面向 K12 课后教学的 Scratch 编程社区，融合「创作 + 社区 + 刷题 + 作业 + 积分 + AI」。

### 核心差异化

1. **sb3 深度解析** — 自动解压 → 解析完整 JSON → 提取程序链/角色/素材/变量/广播 → AI/教师分析
2. **AI 原生** — 主服务内通过 SDK 直接调用 LLM：点评/判题/出题/学情分析
3. **课后教学闭环** — 作业布置 → 提交 → 自动+手动批改 → 数据统计 → 学情报告
4. **班级宠物积分** — 趣味化激励体系

### MVP 边界（v0.3 明确不做）

| 能力 | MVP 状态 | 说明 |
|------|---------|------|
| 积分防刷引擎 | ❌ 简化 | 改用 Redis 计数器 + 简单规则 |
| CDN 三级缓存 | ❌ 简化 | 只用 Nginx + MinIO |
| 数据埋点 | ❌ 不做 | Phase 3+ |
| 内容审核三级流水线 | ❌ 简化 | MVP 只做敏感词匹配 |
| 云变量 WebSocket | ❌ 不做 | Phase 3+ |
| AI Gateway 独立服务 | ❌ 合并 | AI 调用直接在主服务内通过 SDK |
| WebSocket 独立服务 | ❌ 合并 | 使用 Spring WebSocket |
| MongoDB | ❌ 不做 | sb3 解析结果存 MySQL JSON 字段 |
| Elasticsearch | ❌ 不做 | 用 MySQL FULLTEXT + ngram |

## 四、判题引擎设计

### 4.1 双轨判题 — 两级降级（MVP 版）

```
判题请求
  │
  ▼
Step 1: 获取题目 → 选择判题策略
  │
  ├── 选择题/填空题 → ChoiceJudge / FillJudge (直接比对)
  │
  ├── Scratch 算法题:
  │     │
  │     ▼
  │   Level 1: scratch-vm headless (主)
  │     ├── 调用判题沙箱 (Node.js)
  │     ├── 注入测试用例 → 运行 → 捕获输出
  │     ├── 比对结果 → 返回 AC/WA/TLE/MLE/RE
  │     │
  │     ├─ 成功 → 返回结果 ✅
  │     │
  │     └─ 失败/不支持 (画笔/音乐等扩展)
  │          │
  │          ▼
  │       Level 2: 静态分析 (降级)
  │         ├── sb3-parser 中的 StaticAnalyzer
  │         ├── 解析积木结构 → 逻辑验证
  │         └── 返回结构化评分 ✅
  │
  ├── Scratch 创意题:
  │     ├── AI 判题 (主服务内 SDK 调用)
  │     └── 降级: 模板评分
  │
  └── Scratch 混合题 (算法+创意):
        ├── 先 Level 1/2 跑算法部分
        └── 再 AI 评分创意部分
```

> **与 v0.2 对比**：去掉了 Level 3 浏览器沙箱（延后到 Phase 3）。MVP 只有 Level 1 + Level 2，覆盖 90%+ 场景。

### 4.2 判题沙箱 API

```yaml
POST /sandbox/submit
  Request:
    projectUrl: string      # sb3 文件 URL
    problemId: number       # 题目 ID
    testCases: array        # 测试用例 [{input, expectedOutput}]
    timeLimit: number       # ms (默认 30000)
    memoryLimit: number     # MB (默认 512)
  Response:
    taskId: string          # 异步任务 ID

GET /sandbox/result/{taskId}
  Response:
    result: "AC" | "WA" | "TLE" | "MLE" | "RE"
    score: number
    details: [{testCaseId, input, expected, actual, passed, timeUsed}]
    executionLog: string

GET /sandbox/health
  Response:
    status: "ok" | "busy"
    activeTasks: number
    maxTasks: number
```

### 4.3 scratch-vm Headless 执行

```javascript
const vm = new VirtualMachine();
const projectData = fs.readFileSync(projectPath);

// 加载项目
await vm.loadProject(projectData);

// 注入测试用例 (替换"询问并等待"的回答)
vm.runtime.on('QUESTION', (question) => {
  const answer = testCases[currentCase].input;
  vm.runtime.emit('ANSWER', answer);
});

// 运行
vm.greenFlag();

// 等待执行完成或超时
await waitForCompletion(vm, timeLimit);

// 捕获输出
const output = captureOutput(vm);
// output: { variables: {...}, sprites: [...], messages: [...] }
```

## 六、sb3 解析库

### 6.1 解析流程

```
输入: .sb3 文件 (InputStream)

Step 1: SB3Unzipper.unzip(inputStream)
  → project.json (String)
  → assets/ (Map<String, byte[]>)

Step 2: ProjectJsonParser.parse(projectJson)
  → targets: List<Target>

Step 3: 并行提取
  ├── SpriteExtractor   → List<SpriteInfo>
  ├── VariableExtractor → List<VariableInfo>
  ├── BroadcastAnalyzer → List<BroadcastRelation>
  └── BlockChainBuilder → List<ScriptChain>

Step 4: ComplexityCalculator.calculate(allData)
  → ComplexityScore

Step 5: 输出
  → SB3ParseResult (存 MySQL JSON 字段)
  → SB3ParseSummary (轻量摘要，存 MySQL)
```

### 6.2 核心数据结构

```java
@Data
public class SB3ParseResult {
    private String id;
    private ProjectMeta meta;
    private StageInfo stage;
    private List<SpriteInfo> sprites;
    private List<ScriptChain> scripts;
    private List<VariableInfo> variables;
    private List<BroadcastRelation> broadcasts;
    private List<AssetInfo> assets;
    private BlockTypeStats blockStats;
    private ComplexityScore complexity;
}

@Data
public class ScriptChain {
    private String triggerType;    // "greenflag" / "keypress" / "clicked"
    private String spriteName;     // 所属角色
    private BlockNode root;        // 积木树根节点
    private int blockCount;
    private int maxDepth;
    private List<String> blockTypes;
}

@Data
public class ComplexityScore {
    private int totalBlocks;
    private int scriptCount;
    private int maxNestingDepth;
    private int uniqueBlockTypes;
    private int variableCount;
    private int listCount;
    private int broadcastCount;
    private double score;          // 0-100
    private String level;          // "入门" / "基础" / "进阶" / "高级"
}
```

## 七、搜索方案

```
Phase 1-2（MVP）: MySQL FULLTEXT + ngram

works 搜索:
  ALTER TABLE project ADD FULLTEXT INDEX idx_search (title, description) WITH PARSER ngram;
  SELECT * FROM project WHERE MATCH(title, description) AGAINST('关键词' IN BOOLEAN MODE);

problems 搜索:
  ALTER TABLE problem ADD FULLTEXT INDEX idx_search (title, description) WITH PARSER ngram;

users 搜索:
  SELECT * FROM user WHERE nickname LIKE '%关键词%';

优势: 零额外组件，ngram 支持中文分词
限制: 不支持聚合分析、同义词、向量搜索

---

Phase 3+（按需引入 Elasticsearch 8）:

ES 索引设计:
  works_index: title(ik_max_word) / description / tags / user_id / like_count / created_at
  problems_index: title / description / tags / difficulty / ac_rate
  users_index: nickname / bio / level

同步策略: MySQL → Binlog → Canal → ES (近实时同步)
```

```
Spring WebSocket 连接管理
  │
  ├── 连接建立:
  │   用户登录 → 建立 WebSocket → 注册到连接池
  │   心跳保活 (30s 间隔)
  │
  ├── 消息推送:
  │   事件触发 → NotifyService.send(userId, message)
  │     ├── 用户在线 → WebSocket 直接推送
  │     └── 用户离线 → 存入 MySQL (notification 表)
  │
  ├── 通知类型:
  │   ├── 作业通知: "你有一份新作业: xxx，截止时间: xxx"
  │   ├── 批改通知: "你的作业已被批改，得分: 85"
  │   ├── 社交通知: "xxx 点赞了你的作品"
  │   ├── 积分通知: "+20 积分! 你提交了作品"
  │   ├── 宠物通知: "班级宠物升级了! Lv.3"
  │   └── 系统通知: "平台维护公告"
  │
  └── WebSocket 协议:

       连接: ws://host/ws/notify?token=xxx

       消息格式 (服务端→客户端):
       {
         "type": "notification",
         "data": {
           "id": 123,
           "type": "homework_graded",
           "title": "作业已批改",
           "content": "你的作品获得 85 分",
           "data": {"homeworkId": 456},
           "time": "2026-04-23T14:00:00Z"
         }
       }

       心跳: {"type": "ping"} → {"type": "pong"}
```

## 九、积分防刷（简化版）

> MVP 不做完整防刷引擎，用 Redis 计数器 + 简单规则替代。

```
简化防刷规则:

rule 1: 每日积分上限
  └── Redis: SET points:daily:{userId}:{date} (INCR, 上限 200)

rule 2: 点赞唯一性
  └── Redis: SET like:{userId}:{projectId} (SETNX)

rule 3: 操作频率限制
  └── Redis: SET rate:{userId}:{action} (SETNX + TTL 10s)

rule 4: 每日 AI 调用限制
  └── Redis: INCR ai:daily:{userId}:{date} (上限 20)

实现:
  所有规则通过 Redis 原子操作完成，无需独立防刷服务
  异常检测延后到 Phase 3+
```

### 积分规则表

| 行为 | 积分 | 防刷限制 |
|------|------|---------|
| 提交原创作品 | +10 | 同一作品不重复 |
| 作品被点赞 | +2 | 同一用户只计 1 次 |
| 作品被收藏 | +3 | 同一用户只计 1 次 |
| 完成每日一题 | +5 | 每日 1 次 |
| AC 一道题 | +10~50 | 按难度 |
| 按时提交作业 | +15 | 每份作业 1 次 |
| 作业获满分 | +30 | 每份作业 1 次 |
| 获得徽章 | +15 | — |

## 十、内容审核（简化版）

> MVP 不做三级流水线，只做敏感词匹配。

```
用户提交内容 (作品标题/描述/评论)
  │
  ▼
敏感词匹配 (< 10ms)
  ├── AC 自动机匹配敏感词库
  ├── 正则匹配 (手机号/身份证)
  └── 结果: 通过 / 拒绝

存储:
  ├── 敏感词库 → MySQL (sensitive_word 表)
  ├── 敏感词缓存 → Redis (定期刷新)
  └── 审核日志 → MySQL (audit_log 表)

Phase 3+ 扩展:
  └── 接入 AI 审核 (调用 AiClient)
```
