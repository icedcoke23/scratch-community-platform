## 二、Scratch 类平台深度分析

### 2.1 Scratch MIT（scratch.mit.edu）

**平台定位：** 全球最大的 Scratch 创作与分享社区，MIT 终身幼儿园团队出品

#### 核心功能拆解

| 功能模块 | 具体功能 | 实现亮点 |
|----------|---------|---------|
| **创作引擎** | 在线 scratch-gui 编辑器 | 基于 React + Redux，支持实时协作预览 |
| **项目分享** | 一键发布到社区 | 自动截取封面、生成预览动画 |
| **Remix 机制** | 基于任何项目创建副本 | **关键设计：** remix 链自动追溯到原始项目，显示"Based on..." |
| **工作室 (Studio)** | 用户自建项目集合 | 类似歌单/收藏夹，支持多人协作管理 |
| **社区互动** | 点赞 (Love) + 收藏 (Favorite) + 评论 | 双重互动：Love 表达喜爱，Favorite 用于收藏管理 |
| **发现页** | 精选项目 + 趋势 + 编辑推荐 | 多维度展示：最新、最热、最多 remix、编辑精选 |
| **用户主页** | 项目列表 + 关注/粉丝 + 收藏 | 个人品牌展示页 |
| **Scratch 教育版** | 教师创建班级、分配任务 | 面向学校的增强版，支持离线使用 |

#### Remix 机制深度拆解 ⭐

Scratch 的 Remix 是其社区生态的核心引擎：

```
原始项目 A
  ├── Remix 1 (用户X) → "Based on A by 原作者"
  │     ├── Remix 1.1 (用户Y) → "Based on 1 by X (Originally by 原作者)"
  │     └── Remix 1.2 (用户Z)
  └── Remix 2 (用户W)
        └── Remix 2.1 (用户V)
```

**设计精髓：**
- **自动归属**：Remix 项目标题下方始终显示 "Based on [原项目] by [原作者]"
- **不可删除归属**：即使 remix 了 10 层，原始作者信息永不丢失
- **Remix 计数**：每个项目显示被 remix 次数，形成社交货币
- **激励循环**：被 remix 多 = 作品有价值 → 原作者获得成就感 → 继续创作

**数据模型：**
```
projects 表:
  - id
  - parent_id (指向被 remix 的项目，NULL = 原创)
  - root_id (指向最原始的项目，穿透所有层级)
  - remix_count (被 remix 次数)
```

#### 发现与推荐

| 推荐位 | 算法 | 说明 |
|--------|------|------|
| 首页精选 | 编辑人工挑选 | 质量保证 |
| 趋势项目 | 近期互动量加权 | 热度发现 |
| 最新项目 | 时间排序 | 新作曝光 |
| 最多 Remix | Remix 数排序 | 社区影响力 |
| 按标签浏览 | 用户自定义标签 | 垂直发现 |

#### 我们的差距与可借鉴点

| 维度 | Scratch MIT | 我们现状 | 差距 |
|------|------------|---------|------|
| Remix 机制 | 成熟的多层 remix + 归属链 | 已规划基础功能 | 需实现 root_id 穿透 |
| 工作室 | 多人协作项目集 | 未规划 | 可考虑 Phase 3+ |
| 双重互动 | Love + Favorite 分离 | 仅点赞 | 可增加收藏分离 |
| 编辑推荐 | 人工精选 | 已规划教师推荐 | 需扩展为运营推荐 |

└── 参加活动 → +50 积分

积分 → 等级
├── 0-99 → Lv.1 编程小白
├── 100-499 → Lv.2 编程新手
├── 500-1999 → Lv.3 编程达人

## 三、社区/创作者平台深度分析

### 3.1 GitHub

**平台定位：** 全球最大的代码托管与开发者社区

#### 核心机制拆解

| 机制 | 说明 | 对我们的启发 |
|------|------|-------------|
| **Fork** | 代码仓库的完整复制 | 类似 Scratch Remix，但更彻底 |
| **Star** | 收藏/认可，不产生 fork | 区分"喜欢"和"收藏" |
| **Watch** | 关注项目动态 | 关注项目更新通知 |
| **Trending** | 趋势项目，基于 star 增速 | 近期热度而非历史总量 |
| **Topics** | 项目标签分类 | 标签发现机制 |
| **README** | 项目详情页，自定义展示 | 项目详情页可参考 |
| **Contributors** | 贡献者列表 | 多人协作的归属展示 |
| **Insights** | 项目数据分析（star/fork/visit 趋势） | 创作者数据分析 |

#### 推荐算法（Trending）

GitHub Trending 的排序逻辑：
```
热度 = (近7天 star 增量 × 权重1) + (近7天 fork 增量 × 权重2) + (历史 star 基数 × 衰减因子)
```

**关键洞察：** 用"增量"而非"总量"，让新项目有机会被发现

#### 可借鉴点

- **Star vs Love 分离**：GitHub 的 Star 和 Watch 分离，Scratch 的 Love 和 Favorite 分离，都说明"表达喜爱"和"收藏管理"是两个独立需求
- **Trending 增量排序**：用近期增量而非总量，避免马太效应
- **Topics 标签体系**：标准化的标签分类，提升发现效率
- **Insights 数据看板**：为创作者提供项目数据分析

| **创作中心** | UP主数据分析面板 | 创作者工具 |
| **动态** | UP主动态流，类似社交媒体 | 创作者关注流 |

#### 推荐算法特点

- **协同过滤**：看过 A 的人也看了 B
- **标签匹配**：基于用户兴趣标签推荐
- **冷启动**：新内容获得初始曝光池

#### 可借鉴点

- **新作曝光池**：新作品获得初始曝光机会，避免马太效应

## 四、在线评测 OJ 平台深度分析

### 4.1 Codeforces

**平台定位：** 全球最大的算法竞赛平台，俄罗斯团队运营

#### 竞赛系统深度拆解 ⭐

| 功能 | 说明 |
|------|------|
| **比赛分级** | Div.1 / Div.2 / Div.3 / Div.4，按 Rating 分级 |
| **Educational Rounds** | 教学型比赛，侧重特定算法 |
| **Virtual Contest** | 虚拟比赛，任何时候可以参加历史比赛 |
| **Gym** | 训练场，非 Rated 的练习比赛 |
| **Rating 系统** | 基于 Elo 的动态评分，颜色标识实力 |
| **Hack 机制** | 比赛中可以 Hack 他人代码，增加互动性 |
| **Editorial** | 官方题解 + 社区题解 |
| **Tags** | 题目标签（动态规划、图论等） |

#### Rating 系统

```
颜色标识:
  灰色 (Newbie):      0-1199
  绿色 (Pupil):       1200-1399
  青色 (Specialist):  1400-1599
  蓝色 (Expert):      1600-1899
  紫色 (Candidate Master): 1900-2099
  橙色 (Master):      2100-2299
  红色 (Grandmaster):  2300+

Rating 计算:
  基于 Elo 系统，每场比赛后根据排名重新计算
  新用户有较大的波动因子，稳定后波动减小
```

#### 虚拟比赛（Virtual Contest）

**核心价值：** 让任何时候的用户都能体验真实比赛的紧张感

```
虚拟比赛流程:
1. 选择历史比赛
2. 设置开始时间
3. 按真实比赛时间限制答题
4. 排名仅显示参加该虚拟赛的用户
5. 不影响正式 Rating
```

#### 可借鉴点

- **Rating 颜色标识**：直观的实力展示，形成社区身份认同
- **Virtual Contest**：让历史比赛可重复参与，延长竞赛生命周期
- **Hack 机制**：增加互动性和学习深度
- **Educational Rounds**：教学型比赛，侧重学习而非竞技
- **Tags 分类**：精细的题目标签，支持按知识点刷题

### 4.2 LeetCode

**平台定位：** 面试准备 + 算法训练平台

#### 核心功能

| 功能 | 说明 |
|------|------|
| **题目分类** | 按难度(Easy/Medium/Hard) + 按标签(数组/字符串/DP等) |
| **每日一题** | 每天推送一道题，培养刷题习惯 |
| **周赛** | 每周一次竞赛，类似 Codeforces |
| **讨论区** | 题解分享 + 讨论 |
| **学习计划** | 结构化的学习路径（如"30天挑战"） |
| **企业题库** | 按公司分类的面试真题 |
| **Playground** | 在线代码编辑器，支持多语言 |
| **提交统计** | 通过率、用时分布、内存分布 |

#### 判题系统

```
提交流程:
1. 用户提交代码
2. 编译 → 执行 → 比对测试用例
3. 返回结果: Accepted / Wrong Answer / TLE / MLE / RE
4. 显示用时和内存占用
5. 与所有用户对比百分位

测试用例:
- 公开用例：用户可见
- 隐藏用例：提交后显示失败用例
- 边界用例：自动检测边界条件
```

#### 可借鉴点

- **每日一题**：培养用户习惯，提升日活
- **学习计划**：结构化学习路径，降低学习门槛
- **提交统计**：用时/内存对比，增加竞争感
- **讨论区 + 题解**：社区化的学习氛围

## 九、关键数据模型变更建议

### 基于竞品分析的数据库优化

```sql
-- 1. Remix 归属链（借鉴 Scratch MIT）
ALTER TABLE projects ADD COLUMN root_id BIGINT COMMENT '最原始项目ID，穿透所有remix层级';
ALTER TABLE projects ADD COLUMN remix_count INT DEFAULT 0 COMMENT '被remix次数';

-- 2. 收藏与点赞分离（借鉴 GitHub/Pixiv）
-- 已有点赞表，新增收藏表
CREATE TABLE user_favorites (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    project_id BIGINT NOT NULL,
    is_public TINYINT DEFAULT 1 COMMENT '是否公开收藏',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_project (user_id, project_id)
);

-- 3. 竞赛 Rating（借鉴 Codeforces）
ALTER TABLE users ADD COLUMN contest_rating INT DEFAULT 0 COMMENT '竞赛积分';
ALTER TABLE users ADD COLUMN contest_color VARCHAR(20) DEFAULT 'newbie' COMMENT '竞赛颜色标识';

-- 4. 每日排行增量（借鉴 Pixiv/Bilibili）
CREATE TABLE project_daily_stats (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    project_id BIGINT NOT NULL,
    stat_date DATE NOT NULL,
    daily_likes INT DEFAULT 0,
    daily_favorites INT DEFAULT 0,
    daily_remixes INT DEFAULT 0,
    daily_views INT DEFAULT 0,
    UNIQUE KEY uk_project_date (project_id, stat_date)
);

-- 5. 签到系统（借鉴洛谷/编程猫）
CREATE TABLE user_checkins (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    checkin_date DATE NOT NULL,
    consecutive_days INT DEFAULT 1,
    reward_points INT DEFAULT 0,
    UNIQUE KEY uk_user_date (user_id, checkin_date)
);
```
