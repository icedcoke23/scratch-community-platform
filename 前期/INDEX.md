### 架构演进路线
```
v0.1 (13微服务) → 审计发现过度拆分 → v0.2 (1单体+2服务)
```

### v0.2 核心设计要点
- **部署单元 = 3 个**：模块化单体 + 判题沙箱 + AI Gateway
- **sb3 解析**：共享库（非服务），被各模块直接调用
- **判题核心**：统一 JudgeCore 库 + SandboxClient（调用沙箱服务）
- **AI Gateway**：Prompt 管理 / 缓存 / 降级 / 成本控制 / 多模型切换
- **内容审核**：规则引擎 → AI → 人工 三级流水线
- **实时通知**：WebSocket 推送 + 离线存储
- **搜索**：ES 索引（works/problems/users）
- **防刷**：7 条规则引擎
- **CDN**：Nginx 本地 → MinIO → CDN 三级缓存
- **扩展**：JudgePlugin 接口，后续加语言不改核心

## 📌 决策记录

| 日期 | 决策 | 理由 |
|---|---|---|
| 2026-04-23 | 聚焦 Scratch，暂不扩展 | 降低 MVP 复杂度，先把 Scratch 做透 |
| 2026-04-23 | 不做工作室/组队创作 | 简化社区，聚焦核心场景 |
| 2026-04-23 | 教学面向课后 | 作业/测评/班级管理，不含直播录播 |
| 2026-04-23 | sb3 解析是核心差异化 | 自动解压→解析→提取全量信息→AI/教师分析 |
| 2026-04-23 | 基于 teaching-open 起步 | 复用用户/权限/课程/部署等成熟模块 |
| 2026-04-23 | **架构审计：微服务→模块化单体** | 13 服务过度拆分，改为 1 单体 + 2 独立服务 |
| 2026-04-23 | **sb3-parser 从服务改为共享库** | 纯计算逻辑，不需要独立进程 |
| 2026-04-23 | **OJ+作业统一 JudgeCore** | 判题逻辑相同，只是业务场景不同 |
| 2026-04-23 | **AI Gateway 独立服务** | Prompt 管理/缓存/降级/成本控制/多模型切换 |
| 2026-04-23 | **三级内容审核** | 规则引擎→AI→人工，保证内容安全 |
| 2026-04-23 | **7 条防刷规则** | 积分系统需要防作弊 |
| 2026-04-23 | **MVP 存储简化：去掉 MongoDB/ES** | Phase 1-2 仅 MySQL+Redis+MinIO，降低运维复杂度 |
| 2026-04-23 | **判题沙箱三级降级** | Level 1 headless → Level 2 静态分析 → Level 3 浏览器沙箱 |
| 2026-04-23 | **MySQL FULLTEXT 替代 ES** | MVP 阶段数据量小，ngram 够用，Phase 3+ 按需引入 ES |
| 2026-04-23 | **新增竞赛系统 + 用户成长路径** | 功能模块从 10 → 12，Remix/实时排名/成长可视化 |
