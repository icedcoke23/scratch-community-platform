## 1. 整体架构概览

```
┌─────────────────────────────────────────────────────────┐
│                      前端层                              │
│  Vue 3 + Vite + TypeScript + Element Plus / Naive UI    │
│  Scratch GUI (嵌入式编辑器)                               │
├─────────────────────────────────────────────────────────┤
│                      网关层                              │
│  Nginx (反向代理 + 静态资源 + SSL)                        │
├─────────────────────────────────────────────────────────┤
│                      业务层                              │
│  Spring Boot 3 + Java 17 + MyBatis-Plus                │
│  Spring Security + JWT + OAuth2                          │
│  AI 调用 (HTTP SDK 直连外部 API，内置 Prompt 管理 + 降级) │
│  WebSocket (Spring WebSocket，云变量 + 通知推送)          │
├─────────────────────────────────────────────────────────┤
│                   判题沙箱层                              │
│  Level 1: Node.js 18 Headless (scratch-vm)               │
│  Level 2: 静态分析 (Java, sb3-parser)                    │
│  Level 3: 浏览器沙箱 → 延后至 Phase 3                    │
├─────────────────────────────────────────────────────────┤
│                    存储层 (MVP)                          │
│  MySQL 8.0 + Redis 7 + MinIO                             │
└─────────────────────────────────────────────────────────┘
```

**与 v0.3 的核心变化：**
- ❌ 移除 AI Gateway 独立服务（Python FastAPI）→ AI 调用内嵌至 Spring Boot，通过 HTTP SDK 直连外部 API
- ❌ 移除浏览器沙箱（Level 3）→ 延后至 Phase 3
- ✅ WebSocket 方案统一为 Spring WebSocket（不再需要独立服务）
- ✅ 部署单元从 7 个精简至 5 个

## 2. 存储方案 — 分阶段策略

### Phase 1-2（MVP 阶段）

| 组件 | 版本 | 职责 | 说明 |
|------|------|------|------|
| **MySQL 8.0** | 8.0.x | 核心业务数据 + FULLTEXT 搜索 | 使用 ngram parser 支持中文全文搜索，满足 MVP 搜索需求 |
| **Redis 7** | 7.x | 缓存 + 排行榜 + 云变量 + Session + 防刷计数 | Sorted Set 实现排行榜；Hash 存储云变量；String + TTL 实现防刷 |
| **MinIO** | 最新稳定版 | 文件存储（sb3 文件、用户头像、素材等） | S3 兼容 API，本地部署，后续可迁移至云 OSS |

**MVP 阶段不引入的组件：**
- ~~MongoDB~~ — sb3 解析结果在 MVP 阶段量级可控，用 MySQL JSON 字段存储
- ~~Elasticsearch~~ — MySQL FULLTEXT + ngram 足以应对初期搜索需求

### Phase 3+（扩展阶段）

| 组件 | 引入条件 | 职责 |
|------|----------|------|
| **MongoDB 7** | sb3 解析结果量 > 100万条，或文档结构频繁变更 | 存储 sb3 解析后的积木树、项目元数据等半结构化数据 |
| **Elasticsearch 8** | 搜索需求复杂化（多维度筛选、模糊搜索、聚合分析） | 全文搜索、智能推荐、搜索分析 |

### 数据库分工表（MVP）

| 数据类型 | 存储位置 | 理由 |
|----------|----------|------|
| 用户账号/角色/权限 | MySQL | 关系型数据，事务一致性要求高 |
| 项目元数据 | MySQL (JSON 字段) | 结构化 + 半结构化混合，MVP 阶段量小 |
| sb3 文件 | MinIO | 大二进制对象，不适合数据库存储 |
| sb3 解析结果 | MySQL (JSON 字段) | Phase 3+ 量大时迁移至 MongoDB |
| 项目搜索 | MySQL FULLTEXT (ngram) | 中文分词支持，MVP 够用 |
| 排行榜 | Redis Sorted Set | 天然支持，性能优异 |
| 云变量 | Redis Hash | 读写频繁，低延迟要求 |
| 用户 Session | Redis String | 分布式 Session 管理 |
| 防刷计数 | Redis String + TTL | 原子计数 + 过期机制 |
| 素材资源 | MinIO | 图片/音频/视频等静态资源 |
| 竞赛实时排名 | Redis Sorted Set | O(log N) 插入/查询，实时排名场景最优 |
| AI 调用缓存 | Redis String + TTL | 相同输入不重复调用外部 AI API |

## 3. 判题沙箱 — 三级降级方案

### 技术选型

#### Level 1: scratch-vm Headless（首选）

| 项 | 说明 |
|----|------|
| **运行环境** | Node.js 18 LTS |
| **依赖** | jsdom (DOM 模拟) + node-canvas (Canvas 渲染) |
| **Polyfill** | AudioContext mock、Canvas mock、EventTarget mock |
| **优势** | 速度快（毫秒级）、资源占用低、可批量并发 |
| **限制** | 不支持画笔扩展、音乐扩展、视频侦测扩展 |
| **适用场景** | 90% 的编程题判题（逻辑题、循环题、变量题等） |

#### Level 2: 静态分析（降级方案）

| 项 | 说明 |
|----|------|
| **运行环境** | 纯 Java（集成在 sb3-parser 库中） |
| **实现方式** | 解析 sb3 中的 JSON 积木结构 → 构建积木图 → 逻辑验证 |
| **优势** | 无需 Node.js 运行时、零外部依赖、确定性结果 |
| **限制** | 无法验证运行时行为（如动画效果、声音播放），只能验证逻辑结构 |
| **适用场景** | 逻辑正确性验证、代码规范检查、结构化评分 |

#### Level 3: 浏览器沙箱（兜底方案）— ⚠️ 延后至 Phase 3

| 项 | 说明 |
|----|------|
| **状态** | **MVP 不实现，延后至 Phase 3** |
| **运行环境** | Docker 容器: node:18 + Chromium + scratch-gui |
| **控制方式** | Puppeteer 无头浏览器控制 |
| **判题方式** | 运行项目 → 定时截图 → AI 视觉比对（预期截图 vs 实际截图） |
| **优势** | 完整 Scratch 运行环境，支持所有扩展 |
| **限制** | 资源消耗大、速度慢（秒级）、并发能力有限 |
| **延后理由** | Chromium + Puppeteer 增加大量镜像体积和资源消耗；MVP 阶段 Level 1 + Level 2 覆盖 >95% 场景 |

### 降级策略

```
判题请求 → Level 1 (scratch-vm headless)
              │
              ├─ 成功 → 返回结果
              │
              └─ 不支持（含画笔/音乐等扩展）
                   │
                   ▼
              Level 2 (静态分析)
              │
              ├─ 可分析 → 返回结构化评分
              │
              └─ 需要运行时验证
                   │
                   ▼
              Level 3 (浏览器沙箱) ← Phase 3 实现
                   │
                   └─ 返回 AI 比对结果
```

> **MVP 降级兜底**：Level 2 无法分析时，返回"该题型暂不支持自动判题，请联系教师手动批改"，而非等待 Level 3。

## 4. AI 调用方案 — 内嵌 SDK

### 架构

```
Spring Boot App 内部
  │
  ├── AiService (统一入口)
  │   ├── prompt 模板管理 (MySQL 存储 + Redis 缓存)
  │   ├── 请求限流 (Redis 滑动窗口)
  │   ├── 响应缓存 (Redis, TTL 1h)
  │   ├── 成本控制 (每日/每月限额)
  │   ├── 降级策略 (超时/限流/预算耗尽 → 模板回复)
  │   └── HTTP SDK 直连外部 AI API
  │       ├── OpenAI 兼容接口 (通义千问 / DeepSeek / 本地模型)
  │       └── 统一请求/响应封装
  │
  └── 调用方
      ├── JudgeService → AI 创意题评分
      ├── HomeworkService → AI 批改 + 学情分析
      ├── SocialService → AI 作品点评
      └── ContentAuditService → AI 内容审核
```

### 与 v0.3 AI Gateway 的对比

| 维度 | v0.3 (独立 AI Gateway) | v0.4 (内嵌 SDK) |
|------|----------------------|-----------------|
| 部署复杂度 | 需部署 Python + FastAPI + Celery | 零额外部署，复用 Spring Boot |
| 语言栈 | Python (FastAPI) + Java (Spring Boot) | 纯 Java |
| 运维成本 | 高（多语言、多进程） | 低（单体内部模块） |
| 功能 | 完整（队列/缓存/限流/降级） | 等价（用 Spring 生态替代） |
| 扩展性 | 可独立扩缩容 | 随 app 服务一起扩缩容 |
| 适用阶段 | 月活 > 10 万，AI 调用量大 | MVP 阶段，调用量可控 |

**决策：MVP 阶段采用内嵌 SDK 方式，Phase 3+ 若 AI 调用量激增再拆分为独立服务。**

### 降级策略

```
AI 调用 → 检查限流/配额
  │
  ├─ 通过 → HTTP 调用外部 API
  │          │
  │          ├─ 成功 → 返回结果 (缓存)
  │          │
  │          └─ 超时/错误 → 返回模板回复
  │
  └─ 超限 → 返回模板回复
```

## 5. WebSocket 方案 — Spring WebSocket

### 架构

```
Spring Boot App 内部
  │
  ├── WebSocketConfig (配置端点)
  │   └── /ws/cloud-variable  ← 云变量实时同步
  │   └── /ws/notification    ← 通知推送
  │
  ├── CloudVariableHandler
  │   ├── 用户连接 → 注册到 Redis Pub/Sub 频道
  │   ├── 变量变更 → 广播至同项目所有在线用户
  │   └── 断线重连 → 从 Redis 拉取最新状态
  │
  └── NotificationHandler
      ├── 作业通知 / 批改通知 / 社交通知
      ├── 在线用户 → WebSocket 直推
      └── 离线用户 → 存 MySQL notification 表
```

### 与独立 WebSocket 服务的对比

| 维度 | 独立 WebSocket 服务 | Spring WebSocket (内嵌) |
|------|--------------------|-----------------------|
| 部署复杂度 | 需独立部署 Node.js/Elixir 服务 | 零额外部署 |
| 运维成本 | 高 | 低 |
| 扩展性 | 可独立扩缩容 | 随 app 服务一起扩缩容 |
| 功能 | 完整 | 等价（Spring WebSocket + Redis Pub/Sub） |
| 适用阶段 | 大规模并发（>1 万 WebSocket 连接） | MVP 阶段 |

**决策：MVP 阶段采用 Spring WebSocket + Redis Pub/Sub，Phase 3+ 若并发连接数 > 1 万再拆分。**

## 6. 竞赛系统技术方案

| 需求 | Redis 数据结构 | 命令示例 |
|------|---------------|----------|
| 实时排名 | Sorted Set | `ZADD competition:1001 score user_id` |
| 查询 Top N | Sorted Set | `ZREVRANGE competition:1001 0 9 WITHSCORES` |
| 查询用户排名 | Sorted Set | `ZREVRANK competition:1001 user_id` |
| 查询用户分数 | Sorted Set | `ZSCORE competition:1001 user_id` |
| 竞赛状态 | Hash | `HSET competition:1001 status "running" end_time "..."` |
| 参赛者集合 | Set | `SADD competition:1001:players user_id` |

**竞赛积分规则：**
- 限时赛：按完成时间排名，越快积分越高
- 排名赛：按得分排名，支持多轮累计
- 创意挑战赛：评委打分 + 社区投票，加权计算

## 9. 技术决策记录

### 决策 1: 静态分析 vs 运行时判题

| 维度 | 静态分析 (Level 2) | 运行时判题 (Level 1/3) |
|------|--------------------|-----------------------|
| **速度** | 毫秒级 | 秒级（Level 1）/ 数秒（Level 3） |
| **准确性** | 逻辑结构层面，无法验证运行时行为 | 可验证实际运行结果 |
| **资源消耗** | 极低 | 中等（Level 1）/ 高（Level 3） |
| **覆盖范围** | 代码规范、逻辑结构、基本正确性 | 完整运行时行为 |
| **外部依赖** | 无（纯 Java） | Node.js / Docker + Chromium |

**决策：采用三级降级策略，优先运行时判题（Level 1），降级到静态分析（Level 2），浏览器沙箱（Level 3）延后至 Phase 3。**

理由：
- 运行时判题能提供更真实的评测结果
- 静态分析作为快速降级方案，覆盖 Level 1 不支持的场景
- 浏览器沙箱资源消耗大（Chromium 镜像 ~400MB，运行时内存占用高），MVP 阶段 Level 1 + Level 2 覆盖 >95% 场景
- Phase 3 引入 Level 3 时，仅需新增容器，不影响已有架构

### 决策 2: MySQL FULLTEXT vs Elasticsearch

| 维度 | MySQL FULLTEXT (ngram) | Elasticsearch 8 |
|------|----------------------|-----------------|
| **部署复杂度** | 零额外组件 | 需独立集群 |
| **中文搜索** | ngram 分词，基本够用 | IK 分词器，效果更好 |
| **性能** | 百万级数据可接受 | 亿级数据仍优秀 |
| **功能** | 基础全文搜索 | 聚合、suggest、同义词、向量搜索 |
| **运维成本** | 低（已有 MySQL） | 高（独立集群 + 监控） |

**决策：MVP 阶段使用 MySQL FULLTEXT + ngram parser，Phase 3+ 按需引入 Elasticsearch。**

理由：
- MVP 阶段数据量小（预估 < 10 万条项目），MySQL FULLTEXT 性能足够
- ngram parser 支持中文分词，搜索体验可接受
- 减少 MVP 阶段运维复杂度和资源消耗
- 当搜索需求复杂化（智能推荐、聚合分析）时再引入 ES

### 决策 3: AI 调用方式 — 内嵌 SDK vs 独立服务

| 维度 | 内嵌 SDK (Spring Boot 内) | 独立 AI Gateway (Python FastAPI) |
|------|--------------------------|--------------------------------|
| **部署复杂度** | 零额外部署 | 需部署 Python + FastAPI + Celery + Redis |
| **语言栈统一性** | 纯 Java | Java + Python 双语言 |
| **运维成本** | 低 | 高（多进程、多语言、多日志） |
| **Prompt 管理** | MySQL 存储 + Redis 缓存 | 等价 |
| **限流/降级** | Spring 生态（Guava RateLimiter / Redis Lua） | FastAPI 中间件 |
| **响应缓存** | Redis String + TTL | 等价 |
| **独立扩缩容** | ❌ 随 app 服务一起 | ✅ 可独立扩缩容 |
| **AI 调用量瓶颈** | 受 app 服务资源限制 | 可独立处理高并发 |

**决策：MVP 阶段采用内嵌 SDK 方式（Spring Boot 内通过 HTTP SDK 直连外部 AI API），Phase 3+ 若 AI 调用量 > 1 万次/天或需要独立扩缩容，再拆分为独立服务。**

理由：
- MVP 阶段 AI 调用场景有限（创意题评分、作品点评、内容审核），日调用量预估 < 1000 次
- 减少一种语言栈（Python），降低团队学习和运维成本
- Spring 生态完全能覆盖限流、缓存、降级需求
- 拆分成本低：只需将 AiService 抽取为独立 FastAPI 服务，调用方改为 HTTP 调用

### 决策 4: WebSocket 方案 — Spring WebSocket vs 独立服务

| 维度 | Spring WebSocket (内嵌) | 独立 WebSocket 服务 (Node.js/Elixir) |
|------|------------------------|-------------------------------------|
| **部署复杂度** | 零额外部署 | 需独立部署 + 运维 |
| **运维成本** | 低 | 高 |
| **功能完整性** | Spring WebSocket + Redis Pub/Sub | 等价（socket.io / Phoenix） |
| **连接数上限** | 受 JVM 内存限制（约 5000-10000 连接） | 可支持 10 万+ 连接 |
| **广播能力** | Redis Pub/Sub 跨实例广播 | 内置集群支持 |

**决策：MVP 阶段采用 Spring WebSocket + Redis Pub/Sub，Phase 3+ 若并发 WebSocket 连接 > 5000 再拆分为独立服务。**

理由：
- MVP 阶段预估在线用户 < 500，WebSocket 连接数远低于上限
- 云变量同步和通知推送是低频场景，不需要极致性能
- 减少部署单元，降低运维复杂度
- Redis Pub/Sub 已能满足跨实例广播需求
